## 前置知识：自回归采样

LLM 解码时采用自回归采样，过程如下：

1. 用前缀作为输入，模型 forward 输出 logits，转换成概率分布后采样生成下一个 token
2. 将生成的 token 和前缀拼接成新的前缀，重复步骤 1
3. 直到生成 EOS 或达到最大 token 数

```
步骤1: [prompt]           → forward → logits → softmax → 采样 → t1
步骤2: [prompt, t1]       → forward → logits → softmax → 采样 → t2
步骤3: [prompt, t1, t2]   → forward → logits → softmax → 采样 → t3
...每次只生成 1 个 token，串行执行
```

### Softmax 公式

将 logits 转换为概率分布：

```
softmax(x_i) = e^(x_i) / Σ e^(x_j)
```

把每个 logit 取指数，再除以所有指数的总和：

```
logits:  [2.0,  1.0,  0.1]
指数:    [e^2.0, e^1.0, e^0.1] = [7.389, 2.718, 1.105]
总和:    7.389 + 2.718 + 1.105 = 11.212
概率:    [0.659, 0.242, 0.099]   和 = 1.0 ✓
```

两个性质：
- **保序放大**：大的更大，小的更小。指数函数让差距指数级放大
- **保证合法概率**：每个值 ∈ (0,1)，总和 = 1

#### Temperature

在 softmax 之前除以 T，控制分布的"尖锐程度"：

```
logits = [2.0, 1.0, 0.1]

T=1.0 (默认):  [2.0, 1.0, 0.1]  → [0.659, 0.242, 0.099]   正常
T=0.5 (更尖):  [4.0, 2.0, 0.2]  → [0.857, 0.116, 0.027]   集中到最大的
T=2.0 (更平):  [1.0, 0.5, 0.05] → [0.424, 0.257, 0.164]   趋向均匀
T→0:                              → [1.0,   0.0,   0.0]     = argmax
T→∞:                              → [0.333, 0.333, 0.333]   = 均匀分布
```

#### 数值稳定性

实际实现先减去最大值防止溢出，数学上等价：

```python
def softmax(logits):
    x = logits - logits.max()   # 最大的变成 e^0=1，不会溢出
    exp_x = torch.exp(x)
    return exp_x / exp_x.sum()
```

### 常用采样方法

softmax 得到概率分布后，有几种方式选出 token：

| 方法 | 做法 | 特点 |
|------|------|------|
| **Argmax (Greedy)** | 取概率最大的 token | 确定性，零多样性 |
| **Top-k** | 只保留概率最高的 k 个，归一化后采样 | k 控制多样性 |
| **Top-p (Nucleus)** | 保留累积概率达到 p 的最小集合，归一化后采样 | 自适应候选数量 |
| **Temperature** | logits / T 后再 softmax | T>1 更随机，T<1 更确定 |

#### Top-k 详解

```
模型输出 logits（词表大小 151936）:
  [2.1, 0.3, -1.5, 5.8, 0.7, 3.2, -0.4, ...]

第1步: Softmax → 概率分布
  [0.02, 0.003, 0.0001, 0.65, 0.004, 0.05, 0.001, ...]

第2步: 按概率排序，只保留前 k 个（假设 k=3）
  token_3:  0.65
  token_5:  0.05
  token_0:  0.02
  其余 151933 个 → 丢弃

第3步: 重新归一化（让概率重新加到 1）
  总和 = 0.65 + 0.05 + 0.02 = 0.72
  token_3:  0.65 / 0.72 = 0.903
  token_5:  0.05 / 0.72 = 0.069
  token_0:  0.02 / 0.72 = 0.028

第4步: 从归一化后的分布中随机采样
  90.3% 选 token_3, 6.9% 选 token_5, 2.8% 选 token_0
```

k 怎么控制多样性：

```
k=1:       只保留概率最高的 1 个 → 100% 选它 → = argmax
k=50:      保留前 50 个 → 低概率但有创意的词也有机会
k=全词表:  不过滤 → 纯 softmax 采样，最大多样性
```

#### Top-p vs Top-k

Top-k 的问题：固定 k 个，不管概率分布长什么样。

```
场景 A: 模型很确定
  top-1: 0.95, top-2: 0.02, top-3: 0.01 ...
  k=5 → 后 4 个加起来才 3.8%，几乎没用，白保留了

场景 B: 模型很犹豫
  top-1: 0.08, top-2: 0.07, top-3: 0.07, top-4: 0.06 ...
  k=5 → 第 6~20 名概率都差不多，被误杀了

Top-p (p=0.9) 自适应:
  场景 A: top-1 (0.95) 已经 ≥ 0.9 → 只保留 1 个 → 自动变 greedy
  场景 B: 需要保留前 ~15 个才累积到 0.9 → 自动扩大候选集
```

## 为什么 Decode 阶段慢 — Memory-Bound

自回归采样中每次只对 1 个 token 做有效计算，但需要访问全部模型权重和 KV cache，**计算访存比极低**。

以 7B 模型、FP16、bs=1 为例：

```
每步访存量: 加载全部权重 = 7B × 2 bytes = 14 GB
每步计算量: 2 × 7B = 14 GFLOPs

A100:
  HBM 带宽 = 2 TB/s  → 搬运时间 = 14 GB / 2 TB/s = 7 ms
  算力 = 312 TFLOPS   → 计算时间 = 14G / 312T = 0.045 ms

计算访存比 = 14 GFLOPs / 14 GB = 1 FLOP/Byte
A100 平衡点 ≈ 156 FLOP/Byte

→ GPU 算力空转 99%，几乎全在等数据搬运
```

KV cache 读取随序列变长线性增长，进一步加剧访存压力：

```
Qwen3-0.6B, 序列长度 2048:
  KV cache 读取 = 2 × 28层 × 2048 × 8头 × 128维 × 2 bytes ≈ 234 MB/step
```

**GPU 不是算不过来，是数据喂不饱。** 这就是 memory-bound 的本质。

## Speculative Decoding 核心思想

核心洞察：大模型验证 1 个 token 和验证 γ+1 个 token 的**访存量几乎一样**（都要加载一遍全部权重），但后者可以一次确认多个 token。

```
自回归（生成 5 个 token）:
  大模型 forward × 5 次 = 加载权重 5 次 = 5T

投机采样（γ=5, 平均接受 3 个）:
  小模型 forward × 5 次 = 加载小权重 5 次 = 5t（t << T）
  大模型 forward × 1 次 = 加载权重 1 次  = ~T
  总计 ≈ T + 5t ≈ T → 得到 ~4 个 token → 加速 ~4x
```

为什么大模型验证多个 token 不怎么多花时间？因为验证本质上是一次 **prefill**（所有 token 并行计算），而 prefill 是 compute-bound 的，多几个 token 只是多一点计算，访存量不变。

## Vanilla Speculative Decoding

由 Google 和 DeepMind 在 2023 年提出（Leviathan et al. 和 Chen et al.），是所有后续变体的基础。

### 大小模型的关系

Vanilla 方案中的"小模型"是一个**完整但更小的 LLM**，和大模型完全独立，各有各的权重，只是架构类似（都是 Transformer），层数、hidden_size、头数都比大模型少，所以推理更快：

```
大模型（如 Llama-70B）:           小模型（如 Llama-1B）:
  embedding                        embedding
  → 80层 Transformer              → 16层 Transformer
  → LM Head                       → LM Head
  参数: 70B，推理慢                 参数: 1B，推理快 ~70 倍
```

常见的大小模型配对：

| 大模型（target） | 小模型（draft） |
|-----------------|----------------|
| Llama-70B | Llama-7B 或 Llama-1B |
| Qwen3-32B | Qwen3-0.6B |
| CodeLlama-34B | CodeLlama-7B |

这也是 Vanilla 方案的一个缺点 — 需要找一个和大模型"配对"的小模型，而且小模型额外占显存。EAGLE 就是为了解决这个问题，用大模型自身的特征层 + 轻量 head 替代独立小模型。

### 流程

```
第1步: 小模型 Mq 自回归生成 γ 个候选 token
  Mq: [prefix]              → 采样 → t1, 记录 q(t1)
  Mq: [prefix, t1]          → 采样 → t2, 记录 q(t2)
  ...共 γ 次（快，因为小模型小）

第2步: 大模型 Mp 一次 forward 验证
  Mp: [prefix, t1, t2, ..., tγ] → 一次 forward → 得到每个位置的 p(·)

第3步: 从左到右逐个验证
  对每个 ti，比较 p(ti) 和 q(ti)，决定接受或拒绝

第4步:
  全部接受 → 用大模型在 tγ 位置的分布再采样一个 bonus token → 回到第1步
  某个被拒 → 从修正分布重采样该位置，丢弃后续 → 回到第1步
```

### Vanilla 的核心贡献

不在于"用小模型猜"这个做法，而在于**数学上无损的验证框架** —— 最终输出分布和直接用大模型解码**完全一致**。后续所有变体（EAGLE、Medusa 等）都复用了这套 accept/reject 机制，只是换了不同的 draft 策略。

## 大模型验证的完整过程

### 第一步：大模型做一次 forward

假设小模型猜了 γ=4 个 token：`[t1, t2, t3, t4]`

```
大模型输入: [prefix, t1, t2, t3, t4]    ← 拼在一起，一次 forward

大模型输出: 每个位置的概率分布（因为 causal mask，每个位置只看到自己之前的 token）
  位置 prefix末尾 → p(·)   "prefix 之后应该是什么"         → 用来验证 t1
  位置 t1        → p(·)   "prefix+t1 之后应该是什么"       → 用来验证 t2
  位置 t2        → p(·)   "prefix+t1+t2 之后应该是什么"    → 用来验证 t3
  位置 t3        → p(·)   "prefix+...+t3 之后应该是什么"   → 用来验证 t4
  位置 t4        → p(·)   "如果全接受，下一个是什么"        → bonus token
```

一次 forward 同时得到每个位置的独立预测，和逐个自回归的结果完全一样。

### 第二步：逐个比对

对每个候选 token，比较大模型概率 p 和小模型概率 q：

```
验证 t1:
  小模型: q(t1) = 0.6     ← 小模型觉得 t1 有 60% 可能
  大模型: p(t1) = 0.8     ← 大模型觉得 t1 有 80% 可能
  接受概率 = min(1, p/q) = min(1, 0.8/0.6) = min(1, 1.33) = 1.0
  → 100% 接受 ✓（大模型比小模型还认可）

验证 t2:
  小模型: q(t2) = 0.7     ← 小模型很自信
  大模型: p(t2) = 0.3     ← 大模型不太认可
  接受概率 = min(1, p/q) = min(1, 0.3/0.7) = 0.43
  → 掷骰子，43% 概率接受，57% 概率拒绝
```

#### "掷骰子"的含义

"掷骰子"就是生成一个 [0, 1) 之间的均匀随机数 r，和接受概率比较：

```python
r = random.uniform(0, 1)    # 比如 r = 0.35

if r < 0.43:
    accept()                 # r=0.35 < 0.43 → 接受
else:
    reject()                 # 如果 r=0.61 ≥ 0.43 → 拒绝
```

因为 r 均匀分布在 [0, 1)，落在 [0, 0.43) 的概率恰好是 43%，落在 [0.43, 1) 的概率是 57%。

#### 为什么不直接比较 p 和 q 决定接受/拒绝

如果 `p < q` 就直接拒绝，那就变成了**确定性**验证：小模型只要稍微高估一点就被拒，太严格了。

用概率接受的好处：即使大模型觉得这个 token 概率没那么高（p=0.3 < q=0.7），**仍有机会接受**，只是概率降低：

```
p/q 很大（大模型更认可）→ 接受概率高 → 几乎总接受
p/q 接近 1（差不多）    → 接受概率高 → 大概率接受
p/q 较小（大模型不认可）→ 接受概率低 → 大概率拒绝，但偶尔也接受
p/q 极小（严重分歧）    → 接受概率极低 → 几乎总拒绝
```

这是 **rejection sampling**（拒绝采样）的经典做法，它让最终分布精确等于大模型分布 p，而不是某种近似。

### 第三步：处理结果

```
情况 A: 全部接受
  t1 ✓ → t2 ✓ → t3 ✓ → t4 ✓
  收获: 4 个 token + bonus token = 5 个 token

情况 B: 中间被拒
  t1 ✓ → t2 ✗ → t3 丢弃 → t4 丢弃
  收获: 1 个 token (t1) + 从修正分布重新采样 t2' = 2 个 token
```

#### Bonus Token 是什么

大模型做 forward 时，输入是 `[prefix, t1, t2, t3, t4]`，因为 causal mask，每个位置都会产出"下一个 token"的概率分布：

```
大模型输出（5 个位置的概率分布）:
  位置 prefix末尾 → p(·) → 验证 t1 用
  位置 t1        → p(·) → 验证 t2 用
  位置 t2        → p(·) → 验证 t3 用
  位置 t3        → p(·) → 验证 t4 用
  位置 t4        → p(·) → 没有 t5 要验证了，但这个分布已经算出来了！
```

验证只需要前 4 个位置的输出，但位置 t4 的输出是**顺带算出来的**（零额外成本）。它代表的是"看完 prefix+t1+t2+t3+t4 之后，大模型认为下一个 token 应该是什么"。直接从中采样一个 token，就是 bonus token —— 白赚一个。

```
没有 bonus: 猜 4 个全接受 → 收获 4 个 token
有 bonus:   猜 4 个全接受 → 收获 4 + 1 = 5 个 token（多的 1 个零成本）
```

#### 被拒时没有 bonus

```
t1 ✓ → t2 ✗ → t3 丢弃 → t4 丢弃

位置 t1 的输出 p(·) → 用来做修正采样 → 得到 t2'（已经利用了大模型输出）
位置 t2、t3、t4 的输出 → 作废（它们的前提是 t2 被接受，但 t2 被拒了）
```

被拒后，修正采样得到的 token 本身就来自大模型的输出，不需要也不能再额外拿 bonus。

#### 拒绝后具体怎么选

拒绝了小模型猜的 token 之后，**不是**从小模型的其他候选里挑，而是直接从修正分布中采样一个全新的 token


### 拒绝后的修正分布

不是直接从大模型分布 p 采样，而是从修正分布采样，以保证无损：

```
修正分布: norm(max(0, p(x) - q(x)))   对词表中每个 token x
```

#### 为什么不能直接从大模型分布 p 采样

因为接受阶段已经按 min(1, p/q) 的概率"消费"了一部分 p 分布，拒绝后如果再用完整的 p，某些 token 的份额就被重复计算了：

```
词表: {A, B, C}
大模型 p: A=0.3, B=0.5, C=0.2
小模型 q: A=0.7, B=0.2, C=0.1

小模型猜了 A（q(A)=0.7），接受概率 = 0.3/0.7 = 0.43

第一次机会（接受阶段）:
  P(选中 A) = P(猜 A) × P(接受 A) = 0.7 × 0.43 = 0.3
  → A 的目标份额是 p(A)=0.3，已经用完了！

如果第二次机会直接用 p:
  P(选中 A) = P(被拒) × p(A) = 0.57 × 0.3 = 0.171
  P(最终选中 A) = 0.3 + 0.171 = 0.471 ≠ p(A)=0.3 ← 超标了！
```

#### 修正分布的直觉

把 p 分布想象成蛋糕，接受阶段"吃掉"了一部分（= min(p, q)），修正分布是**剩下的部分**：

```
大模型分布 p:  A=0.3, B=0.5, C=0.2    ← 完整的蛋糕
小模型分布 q:  A=0.7, B=0.2, C=0.1

接受阶段"吃掉"的部分 = min(p, q):
  A: min(0.3, 0.7) = 0.3    ← 全吃掉了
  B: min(0.5, 0.2) = 0.2    ← 只吃了一部分
  C: min(0.2, 0.1) = 0.1    ← 只吃了一部分

剩下的 = max(0, p - q):
  A: max(0, 0.3-0.7) = 0    ← A 吃光了，修正分布不选 A
  B: max(0, 0.5-0.2) = 0.3  ← 还剩 0.3，重点补偿
  C: max(0, 0.2-0.1) = 0.1  ← 还剩 0.1

归一化:  A=0, B=0.75, C=0.25
→ 75% 选 B，25% 选 C，不选 A
```

#### 为什么修正分布保证无损 — 严格证明

Speculative Decoding 是数学上可证明无损的，不是启发式近似。

**要证的是：对词表中任意 token x，最终被选中的概率 = p(x)。**

整个过程只有两条路径选中 x：

```
路径 1: 小模型猜了 x，且被接受
路径 2: 小模型猜了别的 token y，被拒绝，修正分布采样到 x
```

**路径 1 的概率**：

$$P_1(x) = q(x) \cdot \min\left(1,\; \frac{p(x)}{q(x)}\right) = \min\bigl(q(x),\; p(x)\bigr)$$

**路径 2 的概率**：

先算总拒绝概率。小模型从词表中采样了某个 token y，被接受的概率是 $q(y) \cdot \min(1, \frac{p(y)}{q(y)})$。其中 $q(y)$ 是小模型选中 y 的概率，$\min(1, \frac{p(y)}{q(y)})$ 是选中后被接受的概率。两者相乘就是"y 被选中且被接受"的联合概率。对所有可能的 y 求和，就是**总接受概率**。1 减去它就是**总拒绝概率**：

$$P(\text{拒绝}) = 1 - \sum_{y} q(y) \cdot \min\left(1,\; \frac{p(y)}{q(y)}\right) = 1 - \sum_{y} \min\bigl(q(y),\; p(y)\bigr)$$

p 和 q 越接近，$\sum \min(p, q)$ 越大，拒绝概率越小。如果 p = q（小模型和大模型完全一致），P(拒绝) = 0，全部接受。

修正分布中 x 的概率。接受阶段对 token x 实际消费了多少概率？

$$P_1(x) = q(x) \cdot \min\left(1,\; \frac{p(x)}{q(x)}\right) = \min\bigl(q(x),\; p(x)\bigr)$$

大模型要求 x 的总概率是 p(x)，接受阶段已经消费了 $\min(p, q)$，还差：

$$p(x) - \min\bigl(q(x),\; p(x)\bigr) = \max\bigl(0,\; p(x) - q(x)\bigr)$$

两种情况：

```
p(A)=0.5, q(A)=0.3  ← 大模型比小模型更看好 A
  接受阶段消费了: min(0.3, 0.5) = 0.3
  大模型要求:     p(A) = 0.5
  还差:           0.5 - 0.3 = 0.2  ← 修正分布要补上

p(B)=0.3, q(B)=0.7  ← 小模型比大模型更看好 B
  接受阶段消费了: min(0.7, 0.3) = 0.3
  大模型要求:     p(B) = 0.3
  还差:           p(B) - p(B) = 0  ← 已经够了，不需要再给
```

所以 $\max(0,\; p(x) - q(x))$ 就是**目标份额 p(x) 减去已消费部分，剩下还需要补多少**。已经够了就是 0，不够就补差额。除以 Z 归一化后变成合法的概率分布（和为 1），就是修正分布 r(x)：

$$r(x) = \frac{\max\bigl(0,\; p(x) - q(x)\bigr)}{Z}, \quad \text{其中} \; Z = \sum_{y} \max\bigl(0,\; p(y) - q(y)\bigr)$$

r(x) 就是**拒绝后从修正分布中采样到 token x 的概率**。它是一个完整的概率分布（和为 1），告诉你拒绝后应该按什么概率选每个 token：

```
词表: {A, B, C}
p: A=0.3, B=0.5, C=0.2
q: A=0.7, B=0.2, C=0.1

max(0, p-q):  A=0, B=0.3, C=0.1
Z = 0 + 0.3 + 0.1 = 0.4

r(A) = 0/0.4   = 0       ← 拒绝后 0% 选 A
r(B) = 0.3/0.4 = 0.75    ← 拒绝后 75% 选 B
r(C) = 0.1/0.4 = 0.25    ← 拒绝后 25% 选 C
                  和=1.0
```

拒绝后就是从这个 r 分布里采样：生成一个随机数，75% 选 B，25% 选 C，不选 A。和 softmax 后的采样方式完全一样，只是概率分布换成了 r。

关键等式 — **修正分布的归一化常数 Z 等于总拒绝概率**：

$$\begin{aligned}
Z &= \sum_{y} \max\bigl(0,\; p(y) - q(y)\bigr) \\
  &= \sum_{y} \bigl[p(y) - \min\bigl(p(y),\; q(y)\bigr)\bigr] \quad \leftarrow \max(0,\; a-b) = a - \min(a,b) \\
  &= \sum_{y} p(y) - \sum_{y} \min\bigl(p(y),\; q(y)\bigr) \\
  &= 1 - \sum_{y} \min\bigl(p(y),\; q(y)\bigr) \\
  &= P(\text{拒绝}) \quad \checkmark
\end{aligned}$$

所以路径 2 的概率：

$$P_2(x) = P(\text{拒绝}) \cdot r(x) = Z \cdot \frac{\max\bigl(0,\; p(x) - q(x)\bigr)}{Z} = \max\bigl(0,\; p(x) - q(x)\bigr)$$

**合计**：

$$P(x) = P_1(x) + P_2(x) = \min\bigl(p(x),\; q(x)\bigr) + \max\bigl(0,\; p(x) - q(x)\bigr)$$

分两种情况验证：

$$\text{当}\; p(x) \geq q(x): \quad \min(p,q) = q(x),\;\; \max(0,\;p-q) = p(x)-q(x),\;\; \text{合计} = q(x) + p(x) - q(x) = p(x) \;\checkmark$$

$$\text{当}\; p(x) < q(x): \quad \min(p,q) = p(x),\;\; \max(0,\;p-q) = 0,\;\; \text{合计} = p(x) + 0 = p(x) \;\checkmark$$

**对任意 p、q 分布，对任意 token x，最终概率都严格等于 p(x)。证毕。**

整个证明只依赖一个恒等式：$\min(a, b) + \max(0,\; a - b) = a$，对任意 $a, b \geq 0$ 成立。这就是为什么 Speculative Decoding 不是"大概对"，而是精确对。不管小模型 q 多烂，最终输出都严格等于大模型分布 p。小模型只影响速度（接受率），不影响质量。

#### min(1, p/q) 的原理 — 与经典 Rejection Sampling 的关系

min(1, p/q) 源自 Rejection Sampling（拒绝采样），但做了关键改造。

**经典 Rejection Sampling**：

```
接受概率 = p(x) / (M × q(x))

其中 M = max_x(p(x)/q(x))，保证接受概率 ≤ 1
拒绝后 → 回到起点重新从 q 采样，循环直到被接受
不需要修正分布，靠反复重试保证最终分布 = p
```

为什么存活率 ∝ p(x)：

```
存活概率 = q(x) × p(x)/(M × q(x)) = p(x)/M

q 被约掉了！每个 token 的存活概率正比于 p(x)
在所有被接受的样本中归一化后恰好等于 p(x)
```

**Speculative Decoding 的改造**：

```
接受概率 = min(1, p(x)/q(x))

没有 M，p/q > 1 时直接 100% 接受
拒绝后 → 不重来，从修正分布 max(0, p-q) 采样，一次搞定
```

为什么不用经典做法：

```
经典做法:
  猜了 token → 大模型拒绝 → 小模型再猜 → 大模型再验证 → 又拒绝 → ...
  每次拒绝都要多一次大模型 forward，完全失去加速意义

Speculative Decoding:
  猜了 token → 大模型拒绝 → 直接从修正分布采样 → 结束
  修正分布用的是大模型本次 forward 已经算出来的 p，零额外 forward
```

min(1, p/q) 的本质：小模型高估了（p < q）就按比例打折接受，没高估（p ≥ q）就全收。牺牲了经典版"q 被完美约掉"的优雅性质，换来"不需要重试"的实用性，再用修正分布补上数学上的缺口。

**前提条件**：

```
1. q(x) > 0 对所有 p(x) > 0 的 x（q 的支撑集覆盖 p）
   → LLM 中天然满足，softmax 输出永远 > 0
2. 大模型和小模型用相同的采样参数（temperature、top-k 等）
   → 工程实现需要保证
```

#### 小模型的完整概率分布从哪来

小模型生成 token 时，softmax 算出的就是**整个词表**的概率分布，只是最终只采样了一个 token，但完整分布一直在内存里：

```python
# 伪代码：小模型生成阶段
draft_tokens = []
draft_probs = []

for i in range(γ):
    logits = draft_model(prefix)         # forward → logits [vocab_size]
    q = softmax(logits)                   # 完整概率分布 [vocab_size]
    token = sample(q)                     # 只采样一个 token

    draft_tokens.append(token)            # 保存选中的 token
    draft_probs.append(q)                 # 保存完整概率分布 ← 留给验证用

    prefix = concat(prefix, token)

# 验证阶段
for i in range(γ):
    p_i = target_probs[i]                 # 大模型在位置 i 的分布
    q_i = draft_probs[i]                  # 小模型在位置 i 的分布（之前保存的）
    token = draft_tokens[i]
    accept_prob = min(1, p_i[token] / q_i[token])
    # ...
```

不需要额外计算，生成时本来就算出来了，只需多存一份。

### 完整数值例子

```
词表: {猫, 狗, 鱼}
γ = 2（小模型猜 2 个 token）

═══ 小模型自回归生成 ═══

位置 1: q 分布 = {猫:0.5, 狗:0.3, 鱼:0.2}  → 采样得到 "猫"  q(猫)=0.5
位置 2: q 分布 = {猫:0.1, 狗:0.6, 鱼:0.3}  → 采样得到 "狗"  q(狗)=0.6

候选序列: [猫, 狗]

═══ 大模型一次 forward ═══

输入: [prefix, 猫, 狗]
输出:
  位置 prefix → p = {猫:0.8, 狗:0.1, 鱼:0.1}    验证 "猫"
  位置 猫     → p = {猫:0.2, 狗:0.2, 鱼:0.6}    验证 "狗"
  位置 狗     → p = {猫:0.3, 狗:0.4, 鱼:0.3}    bonus token

═══ 验证 ═══

验证 "猫": p(猫)/q(猫) = 0.8/0.5 = 1.6 → min(1, 1.6) = 1.0 → 接受 ✓

验证 "狗": p(狗)/q(狗) = 0.2/0.6 = 0.33 → 掷骰子
  如果 < 0.33 → 接受 ✓ → 从位置狗的分布采样 bonus token
  如果 ≥ 0.33 → 拒绝 ✗ → 修正分布重采样:
    max(0, p-q) = {猫:0.1, 狗:0, 鱼:0.3} → 归一化 → {猫:0.25, 狗:0, 鱼:0.75}
    → 75% 选 "鱼"，25% 选 "猫"
```

## EAGLE

EAGLE（Extrapolation Algorithm for Greater Language-model Efficiency）来自北大团队，核心思想：**用大模型自身的特征层输出训练一个轻量 draft head，跳过完整的小模型推理**。

### 与 Vanilla 的区别

```
Vanilla:
  独立小模型: embedding → 24层 Transformer → LM Head → token
  大模型: 验证

EAGLE:
  大模型 forward 产出 hidden_states（本来就有，零额外计算）
       ↓
  EAGLE head 输入: embed(token) + hidden_state
       ↓
  EAGLE head: 1层 Transformer（轻量） → 预测下一个 hidden_state
       ↓
  复用大模型的 LM Head → token
       ↓
  大模型: 验证
```

### 为什么更好

**1. Draft 质量更高**

Vanilla 的小模型和大模型是独立训练的，"思考方式"不同，猜测准确率有上限。EAGLE 直接在大模型的特征空间中工作，draft 和 target 天然对齐，接受率更高（~75% vs ~60%）。

**2. 极低开销**

EAGLE head 只有 1 层 Transformer + 一个 FC 层，参数量是大模型的 ~1-2%。复用大模型的 LM Head 权重，不需要额外存储词表投影。

**3. 无需独立小模型**

不需要找配对的小模型，不占额外显存加载第二套权重。

### 训练

用大模型在训练数据上跑 forward，收集 hidden_states 作为训练数据：

```
输入:  embed(token[t+1]) + hidden_state[t]  (token embedding + 大模型倒数第二层在位置 t 的输出)
标签:  hidden_state[t+1]
损失:  预测的 hidden_state 经过 LM Head 后与真实 next token 的交叉熵
```

为什么需要两部分输入：
- hidden_state[t] 包含前文语义，但不知道最终采样了哪个 token（它产出的是概率分布）
- embed(token[t+1]) 显式告诉 EAGLE head "实际选了哪个 token"，才能基于此预测下一步

训练成本很低，通常几小时 GPU 时间。

### 为什么可以预测下一个 Hidden State

核心假设：**hidden state 之间的变化是有规律可学的**。

```
观察: 大模型最后一层的 hidden state 序列

h0, h1, h2, h3, h4, ...

这些向量不是随机的，它们是同一个模型对连续文本的内部表示，
相邻位置之间存在强烈的相关性和可预测的变化模式。
```

三个原因：

```
原因 1: Transformer 的 attention 天然在做"汇聚上下文"

  h[t] 不只是 token[t] 的表示，而是"看完 token[0..t] 后的总结"
  这个总结里已经包含了对未来的预期
  例: h("今天天气") 里隐含了"下一个可能是 好/不好/怎么样"的预期

原因 2: 语言本身有强局部相关性

  连续 token 的 hidden state 变化通常是平滑的（渐变），不是跳跃的
  只有在语义转折时才会剧变
  → 一个轻量网络就能捕捉这种平滑变化

原因 3: 实验验证

  EAGLE 论文实测: 1 层 Transformer 就能达到 70-80% 的接受率
  说明大部分情况下，h[t] → h[t+1] 的变化确实是可预测的
```

### 与 MTP（Multi-Token Prediction）的关系

EAGLE 和 MTP 的核心假设一样：**hidden state 包含关于未来的信息**。

```
MTP (DeepSeek-V3 等的训练方法):
  训练时: 一个 hidden state 不只预测下一个 token，还预测后面 2-4 个 token
  目的:   让 hidden state 编码更多未来信息，提升模型质量
  额外结构: 多个预测 head，每个预测不同距离的 future token

EAGLE:
  推理时: 用轻量 head 从当前 hidden state 预测下一个 hidden state
  目的:   快速生成 draft token，加速推理
  额外结构: 1 层 Transformer head
```

| | MTP | EAGLE |
|---|---|---|
| **阶段** | 训练 | 推理 |
| **目的** | 让模型学得更好 | 让推理更快 |
| **做法** | 用 hidden state 直接预测 future token | 用 hidden state 预测 future hidden state |
| **核心假设** | hidden state 包含未来信息 | hidden state 的变化可预测 |
| **额外结构** | 多个 MTP head | 1 个 EAGLE head |

**MTP 训练的模型天然更适合 EAGLE**：MTP 显式训练 hidden state 编码未来多个 token 的信息，EAGLE 在这种模型上 draft 质量更高、接受率更高。

## EAGLE-2

在 EAGLE 基础上引入**动态 draft tree**：

### EAGLE-1 的问题

```
EAGLE-1 固定猜 γ 个，串成一条链:
  h0 → h1' → h2' → h3' → h4' → h5'
                                  ↑ 误差累积，后面越来越不准

问题:
  1. 第 2 个位置有两个很可能的候选（比如"好"和"棒"），但只能选一个
  2. 后面即使不准也必须猜完 γ 个，浪费计算
  3. 猜得很准时也不敢多猜，γ 是固定的
```

### 动态 Draft Tree

```
EAGLE-1 (链):
  t1 → t2 → t3 → t4 → t5       只有一条路

EAGLE-2 (树):
           t1
          / \
        t2a  t2b              ← t1 位置有两个高置信候选，都展开
        |     |
       t3a   t3b
        |
       t4a                    ← t3a 置信度高，继续展开
                              ← t3b 置信度低，停止
```

### 怎么决定展开哪些分支

EAGLE head 预测 hidden state 后经过 LM Head 得到概率分布，这个分布本身就带有置信度信息：

```
位置 1 的 draft 分布:
  "好": 0.45
  "棒": 0.35
  "不": 0.05
  ...

策略: 概率 > 阈值 θ 的候选都保留
  θ = 0.2 → 保留 "好"(0.45) 和 "棒"(0.35) → 展开 2 个分支
  θ = 0.4 → 只保留 "好"(0.45) → 展开 1 个分支
```

或者用 **top-k 展开**：每个位置保留概率最高的 k 个候选。k 可以根据置信度动态调整：

```
分布很集中（top-1 概率 0.9）→ k=1，一个就够
分布很分散（top-1 概率 0.3）→ k=2 或 3，多试几个
```

### 树形验证：大模型怎么一次验证一棵树

关键：树上所有节点可以**拼成一个 batch**，大模型一次 forward 验证整棵树。

```
树结构:
           t1
          / \
        t2a  t2b
        |     |
       t3a   t3b

拼成输入（用特殊的 attention mask）:
  路径 1: [prefix, t1, t2a, t3a]
  路径 2: [prefix, t1, t2b, t3b]

attention mask 保证:
  t2a 只能看到 [prefix, t1]        ← 不能看到 t2b
  t3a 只能看到 [prefix, t1, t2a]   ← 不能看到 t2b, t3b
  t3b 只能看到 [prefix, t1, t2b]   ← 不能看到 t2a, t3a

→ 每个节点的 attention 只看自己路径上的祖先，和自回归一致
→ 一次 forward 得到所有节点的 p 分布
```

```
具体实现: tree attention mask

位置:    prefix  t1  t2a  t2b  t3a  t3b
prefix:    1     0    0    0    0    0
t1:        1     1    0    0    0    0
t2a:       1     1    1    0    0    0    ← t2a 看 prefix + t1
t2b:       1     1    0    1    0    0    ← t2b 看 prefix + t1（不看 t2a）
t3a:       1     1    1    0    1    0    ← t3a 看 prefix + t1 + t2a
t3b:       1     1    0    1    0    1    ← t3b 看 prefix + t1 + t2b
```

### 验证后选最长被接受的路径

```
大模型验证结果:
  t1  → 接受 ✓
  t2a → 接受 ✓     t2b → 拒绝 ✗
  t3a → 接受 ✓     t3b → (父节点 t2b 已拒绝，跳过)

选最长接受路径: t1 → t2a → t3a（3 个 token + bonus）
比 EAGLE-1 的链可能只接受 t1 → t2（2 个）更多
```

### 为什么树比链好

```
链: 只押一条路，错了就断
  t1(对) → t2(错) → t3(丢弃) → t4(丢弃)
  收获: 1 个 token

树: 多条路并行，一条断了还有别的
  t1(对) → t2a(错)  → t3a(丢弃)
         → t2b(对)  → t3b(对)
  收获: 3 个 token（走 t1→t2b→t3b 这条路）
```

### 树的开销

```
链（γ=5）: EAGLE head forward 5 次，大模型验证 5 个 token
树（展开 10 个节点）: EAGLE head forward 10 次，大模型验证 10 个 token

树的计算量更大，但:
  1. EAGLE head 很轻（1 层 Transformer），多算几次开销小
  2. 大模型验证是 prefill 模式（compute-bound），多几个 token 影响不大
  3. 接受的 token 更多，抵消了额外开销

净效果: 接受率从 ~75% 提升到 ~80%，加速比从 2.5-3x 提升到 3-4x
```

### EAGLE-1 vs EAGLE-2

| | EAGLE-1 | EAGLE-2 |
|---|---|---|
| Draft 结构 | 固定链 | 动态树 |
| 每轮 draft 数 | 固定 γ 个 | 动态，按置信度展开 |
| 分支策略 | 无，每个位置只选 1 个 | 高置信多展开，低置信早停 |
| 验证方式 | 顺序验证链 | tree attention 一次验证整棵树 |
| 接受率 | ~75% | ~80% |
| 加速比 | 2.5-3x | 3-4x |
| 实现复杂度 | 简单 | 需要 tree attention mask |

## N-gram 作为 Draft Model

Speculative Decoding 的 draft model 不一定是神经网络，**N-gram 查表也可以**。

### 做法

从 prompt 或已生成的文本中统计 N-gram 频率，用来预测接下来的 token：

```
已有文本: "The quick brown fox jumps over the lazy"
4-gram 查找: 在文本中找到 "the lazy" 后面跟过 "dog" → 猜测 "dog"
```

### 优势

| 特性 | 为什么适合做 draft |
|------|------------------|
| 极快 | 纯查表，不需要 GPU，延迟 ~微秒级 |
| 零显存 | 不需要加载额外模型权重 |
| 高命中率（特定场景） | 翻译、摘要、代码补全等重复多的任务 |

### 局限

命中率高度依赖文本的**重复性**。开放问答、创意写作等重复少的场景，N-gram 猜不准，接受率低。通常作为补充策略，和神经网络 draft model 配合使用。

### 典型实现

- Google REST（Retrieval-based Speculative Decoding）：从 prompt 本身做 N-gram 匹配
- llama.cpp `--draft` 模式：支持 N-gram lookup

## 各方案对比总结

| 方案 | Draft 策略 | 加速比 | 接受率 | 额外显存 |
|------|-----------|--------|--------|----------|
| **Vanilla** | 独立小模型自回归猜 K 个 | 1.5-2x | ~60% | 需加载小模型 |
| **Medusa** | 大模型 + 多个并行 head | 2-2.5x | ~65% | 额外训练的 head |
| **EAGLE** | 大模型特征层 + 轻量 head | 2.5-3x | ~75% | 极小（~1-2%） |
| **EAGLE-2** | EAGLE + 动态 tree | 3-4x | ~80% | 极小 |
| **N-gram** | 文本查表 | 场景依赖 | 场景依赖 | 几乎为零 |

```
演进路线:

Vanilla:    小模型 → 串行猜 K 个 → 大模型一次验证
                     ↓ 去掉独立小模型
Medusa:     大模型 + 多个 head → 并行猜多位置 → 大模型验证
                     ↓ 在特征空间中 draft
EAGLE:      大模型特征层 + 轻量 head → draft → 大模型验证
                     ↓ 动态展开
EAGLE-2:    EAGLE + 动态 tree → 自适应展开

N-gram:     独立路线，纯查表，零开销，适合高重复场景
```

所有方案共享 Vanilla 提出的**无损验证框架**：`min(1, p/q)` 接受 + `max(0, p-q)` 修正采样，保证最终输出分布和纯用大模型解码完全一致。差异只在 draft 策略。

## 接受率分析与自适应部署策略

### 各方案的典型接受率

| 方案 | 典型接受率 | 每轮平均接受 token 数（γ=5） |
|------|-----------|---------------------------|
| **Vanilla**（配对小模型） | 50-65% | 2.5-3.2 |
| **Medusa** | 60-70% | 3.0-3.5 |
| **EAGLE** | 70-80% | 3.5-4.0 |
| **EAGLE-2**（动态树） | 75-85% | 3.8-4.3 |
| **N-gram** | 30-90%（场景依赖极大） | 1.5-4.5 |

### 任务类型对接受率的影响

不同任务的"可预测性"差异很大，直接影响 draft 的命中率：

| 任务类型 | 接受率趋势 | 原因 |
|---------|-----------|------|
| 代码补全 | 高（70-90%） | 语法模式固定，变量名可预测 |
| 翻译 | 中高（60-80%） | 对照原文，候选空间较小 |
| 摘要 | 中（50-70%） | 需要压缩，但内容来自原文 |
| 多轮对话 | 中（50-70%） | 上下文约束较强 |
| 开放问答 | 中低（40-60%） | 答案空间大 |
| 创意写作 | 低（30-50%） | 高度发散，small model 难猜 |

### 接受率与加速比的关系

设每轮猜 γ 个 token，接受率为 α，大模型单步耗时 T，小模型单步耗时 t：

```
普通自回归: 生成 1 个 token 需要 T

投机采样: 每轮耗时 = γ × t + T（小模型猜 γ 次 + 大模型验证 1 次）
         每轮收获 = α × γ + 1 个 token（期望接受数 + 修正/bonus token）

加速比 = (α × γ + 1) × T / (γ × t + T)
```

数值例子（γ=5, t=T/20, 即小模型比大模型快 20 倍）：

```
α=80%:  加速比 = (4+1) × T / (5×T/20 + T) = 5T / 1.25T = 4.0x ✓
α=60%:  加速比 = (3+1) × T / (5×T/20 + T) = 4T / 1.25T = 3.2x ✓
α=40%:  加速比 = (2+1) × T / (5×T/20 + T) = 3T / 1.25T = 2.4x
α=20%:  加速比 = (1+1) × T / (5×T/20 + T) = 2T / 1.25T = 1.6x
α=0%:   加速比 = (0+1) × T / (5×T/20 + T) = T / 1.25T  = 0.8x ← 变慢了！
```

**当接受率太低时，投机采样反而比普通自回归慢**，因为小模型的 γ 次 forward 白跑了，还额外花了时间。

### 什么时候投机采样会变慢

```
变慢条件: 加速比 < 1
即: (α × γ + 1) × T < γ × t + T
化简: α × γ < γ × t/T
     α < t/T

当接受率 α 低于 小模型/大模型 的耗时比 t/T 时，投机采样就是负收益。
```

例子：

```
t/T = 1/20 = 5%   → α < 5% 才会变慢（几乎不可能，很安全）
t/T = 1/5  = 20%  → α < 20% 就会变慢（小模型不够小，风险增大）
t/T = 1/2  = 50%  → α < 50% 就会变慢（小模型太大了，不值得）
```

**核心原则：小模型必须足够快（t/T 足够小），否则猜错的代价太大。**

### 自适应策略

#### 1. 动态 γ 调整

根据运行时的接受率动态调整每轮猜测数量：

```
初始: γ = 5
每 N 轮统计一次接受率 α

if α > 0.8:
    γ = min(γ + 1, γ_max)     # 猜得准，多猜一点
elif α < 0.4:
    γ = max(γ - 1, 1)          # 猜不准，少猜一点
else:
    γ 不变
```

直觉：如果 draft model 每轮 5 个都能中 4 个，说明它很靠谱，可以尝试猜 6 个、7 个。如果只中 2 个，不如少猜几个减少浪费。

#### 2. 退化保护

当接受率持续过低时，自动退回普通自回归：

```
if 连续 K 轮接受率 < threshold:
    切换到普通自回归解码
    每隔 M 步尝试一次投机采样
    if 接受率回升:
        恢复投机采样模式
```

#### 3. 混合 Draft 策略

不同阶段用不同的 draft 方法：

```
生成开头（高不确定性）→ 用 EAGLE（接受率最高）
生成中段（模式稳定）  → 用 N-gram（零开销，速度最快）
生成结尾（可预测性高）→ 用 N-gram 或加大 γ
```

### 部署场景推荐

| 场景 | 推荐策略 | 原因 |
|------|---------|------|
| **在线服务，低并发** | EAGLE/EAGLE-2, γ=5-7 | GPU 空闲算力多，投机采样收益大 |
| **在线服务，高并发** | 不用或小 γ=2-3 | batch 大时 GPU 已接近满载，投机采样反而争抢资源 |
| **离线批处理** | 不用 | 大 batch + continuous batching 已经充分利用 GPU |
| **单用户交互** | EAGLE-2 + 动态 γ | 延迟敏感，GPU 完全空闲，收益最大 |
| **代码补全** | EAGLE 或 N-gram | 接受率高，N-gram 也很有效 |
| **长文档生成** | EAGLE + 混合策略 | 前段用 EAGLE，后段用 N-gram |

### 为什么高并发时收益下降

投机采样的本质是**用空闲算力换延迟**。Memory-bound 的 decode 阶段 GPU 算力空转 99%，投机采样利用这些空转的算力并行验证多个 token。

但高并发时，batch size 大，GPU 的算力逐渐被填满：

```
bs=1:   计算访存比 = 1 FLOP/Byte     GPU 空转 99% → 投机采样收益大
bs=32:  计算访存比 = 32 FLOP/Byte    GPU 空转 ~80% → 仍有收益
bs=128: 计算访存比 = 128 FLOP/Byte   GPU 接近满载 → 投机采样增加的计算成为瓶颈
bs=256: 计算访存比 > 平衡点          GPU 已满载 → 投机采样纯负收益
```

**投机采样是 bs=1（或小 batch）场景下的利器，不是万能加速方案。**
