CUDA 编程基础

1.1 GPU 硬件架构：SM、HBM、SRAM

在写 CUDA 代码之前，先理解 GPU 的硬件结构。以 NVIDIA A100 为例：

GPU（一块显卡） ┌──────────────────────────────────────────────────────────┐ │ │ │ SM 0 SM 1 SM 2 ... SM 107 │ │ ┌─────────┐ ┌─────────┐ ┌─────────┐ ┌─────────┐ │ │ │ CUDA核心 │ │ CUDA核心 │ │ CUDA核心 │ │ CUDA核心 │ │ │ │ (计算单元)│ │ (计算单元)│ │ (计算单元)│ │ (计算单元)│ │ │ │ │ │ │ │ │ │ │ │ │ │ SRAM │ │ SRAM │ │ SRAM │ │ SRAM │ │ │ │ 192KB │ │ 192KB │ │ 192KB │ │ 192KB │ │ │ └─────────┘ └─────────┘ └─────────┘ └─────────┘ │ │ │ │ ┌────────────────────────────────────────────────────┐ │ │ │ HBM（显存） │ │ │ │ 80 GB，带宽 2 TB/s │ │ │ │ 所有 SM 共享，但访问慢（~400 周期） │ │ │ └────────────────────────────────────────────────────┘ │ └──────────────────────────────────────────────────────────┘
硬件全称类比 CPU特点
SM Streaming Multiprocessor CPU 的一个核 GPU 的基本计算单元。一个 GPU 有几十~上百个 SM。一个 block 只能运行在一个 SM 上，但一个 SM 可以同时运行多个 block。
SRAM Static RAM（Shared Memory） CPU 的 L1 Cache 每个 SM 私有，快但小（~192KB）。访问延迟 ~20 周期。同一 block 内的线程通过它共享数据。
HBM High Bandwidth Memory（显存） CPU 的主存 (DRAM) 所有 SM 共享，大但慢（80GB，但访问延迟 ~400 周期）。数据从 CPU 拷过来就放在这里。
性能瓶颈的根源
SRAM 比 HBM 快 ~20 倍，但容量小 3 个数量级。CUDA 优化的核心思路就是：尽量从 SRAM 读数据，减少 HBM 访问。后面的 Tiling GEMM 和 FlashAttention 都是这个思路。

1.2 编程模型

CUDA 程序 = Host 代码（CPU） + Device 代码（GPU）。CPU 负责控制流，GPU 负责并行计算。

CPU（Host） GPU（Device） ┌──────────────┐ ┌──────────────────────┐ │ 1. 分配显存 │ ──cudaMalloc──→ │ 显存（HBM） │ │ 2. 拷贝数据 │ ──cudaMemcpy──→ │ 数据就位 │ │ 3. 启动 kernel │ ──kernel<<<>>>─→ │ 成千上万线程并行执行 │ │ 4. 等待完成 │ ←─同步───────── │ 计算完毕 │ │ 5. 拷回结果 │ ←─cudaMemcpy── │ │ │ 6. 释放显存 │ ──cudaFree───→ │ │ └──────────────┘ └──────────────────────┘

1.3 线程层次与 Warp

CUDA 的线程组织是四层结构（软件三层 + 硬件一层）：

Grid（网格）—— 一次 kernel launch 产生一个 grid ├── Block 0（线程块）—— 调度到某个 SM 上运行 │ ├── Warp 0（硬件调度单元）—— 32 个线程锁步执行 │ │ ├── Thread 0 ~ 31 │ ├── Warp 1 │ │ ├── Thread 32 ~ 63 │ └── ... ├── Block 1 │ ├── Warp 0 │ │ ├── Thread 0 ~ 31 │ └── ... └── ...
  • Grid：所有线程块的集合，可以是 1D/2D/3D
  • Block：一组线程，同一 block 内的线程共享 SRAM（shared memory），可以同步。一个 block 被分配到一个 SM 上运行
  • Warp：GPU 真正的执行单位。硬件自动把 block 内的线程按 32 个一组分成 warp。同一 warp 的 32 个线程在同一时刻执行相同的指令（SIMT: Single Instruction, Multiple Threads）
  • Thread：最小执行单元，每个线程有唯一 ID

1.4 Warp 深入理解

为什么 block_size 通常是 32 的倍数？
因为硬件以 warp（32 线程）为单位调度。如果 block_size = 100，最后一个 warp 只有 4 个活跃线程，其余 28 个空转浪费算力。所以通常选 128、256、512 等 32 的倍数。

一个 block 有 256 个线程时，硬件会拆成 8 个 warp：

Block（256 个线程） 运行在 SM 上 ┌──────────────────────────────────────────┐ │ Warp 0: thread 0 ~ 31 ← 32 个线程锁步 │ │ Warp 1: thread 32 ~ 63 │ │ Warp 2: thread 64 ~ 95 │ │ Warp 3: thread 96 ~ 127 │ │ Warp 4: thread 128 ~ 159 │ │ Warp 5: thread 160 ~ 191 │ │ Warp 6: thread 192 ~ 223 │ │ Warp 7: thread 224 ~ 255 │ └──────────────────────────────────────────┘ SM 的 warp 调度器在这些 warp 之间快速切换： Warp 0 在等 HBM 数据 → 切到 Warp 3 执行计算 Warp 3 执行完一条指令 → 切到 Warp 5 ...（零开销切换，隐藏内存延迟）
Warp Divergence（分支分歧）
同一 warp 的 32 个线程必须执行相同指令。如果代码中有 if-else，部分线程走 if、部分走 else，硬件会串行执行两个分支，走 if 时 else 的线程空等，反之亦然。性能最差可能减半。

1.5 软件概念 vs 硬件概念

层级软件/硬件你在代码里控制？运行在哪？
Grid软件是（<<>>）整个 GPU
Block软件是（<<<..., block_size>>>）一个 SM
Warp硬件否，硬件自动按 32 分组SM 的 warp 调度器
Thread软件是（threadIdx）SM 的 CUDA 核心
一句话总结调度
你写代码定义 Grid 和 Block → GPU 把 Block 分配到 SM → SM 把 Block 内线程按 32 个一组分成 Warp → Warp 调度器在多个 Warp 之间快速切换来隐藏延迟。

线程的全局唯一 ID 计算：

// 1D 情况
int global_id = blockIdx.x * blockDim.x + threadIdx.x;

// 2D 情况（矩阵计算常用）
int row = blockIdx.y * blockDim.y + threadIdx.y;  // 行
int col = blockIdx.x * blockDim.x + threadIdx.x;  // 列

1.7 第一个完整 CUDA 程序：数组元素翻倍

把 Kernel 函数和内存管理放在一起，写一个可直接编译运行的完整示例。功能：把数组每个元素乘以 2。

// double_array.cu
// 编译: nvcc -o double_array double_array.cu
// 运行: ./double_array

#include <stdio.h>
#include <cuda_runtime.h>

// =============================================
// Kernel: 每个线程把一个元素乘以 2
// __global__ 修饰 = CPU 调用、GPU 执行
// =============================================
__global__ void double_array(float *data, int n) {
    int idx = blockIdx.x * blockDim.x + threadIdx.x;
    if (idx < n) {           // 边界检查！线程总数可能 > 数据量
        data[idx] *= 2.0f;
    }
}

int main() {
    int n = 1024;
    size_t size = n * sizeof(float);

    // === 1. CPU 侧：分配内存并初始化 ===
    float *h_data = (float *)malloc(size);  // h_ 前缀 = host（CPU）
    for (int i = 0; i < n; i++) h_data[i] = (float)i;  // [0, 1, 2, ..., 1023]

    // === 2. GPU 侧：分配显存（HBM）===
    float *d_data;                        // d_ 前缀 = device（GPU）
    cudaMalloc(&d_data, size);

    // === 3. 把数据从 CPU 拷到 GPU ===
    cudaMemcpy(d_data, h_data, size, cudaMemcpyHostToDevice);

    // === 4. 启动 Kernel ===
    int block_size = 256;                          // 每个 block 256 个线程（8 个 warp）
    int grid_size = (n + block_size - 1) / block_size;  // 向上取整 = 4 个 block
    double_array<<<grid_size, block_size>>>(d_data, n);
    //           ↑ 4 个 block   ↑ 每个 256 线程   → 共 1024 个线程，刚好覆盖数组

    // === 5. 把结果从 GPU 拷回 CPU ===
    cudaMemcpy(h_data, d_data, size, cudaMemcpyDeviceToHost);

    // === 6. 验证结果 ===
    printf("h_data[0]   = %.1f (expected 0.0)\n",   h_data[0]);
    printf("h_data[1]   = %.1f (expected 2.0)\n",   h_data[1]);
    printf("h_data[100] = %.1f (expected 200.0)\n", h_data[100]);
    printf("h_data[1023]= %.1f (expected 2046.0)\n",h_data[1023]);

    // === 7. 释放内存 ===
    cudaFree(d_data);   // 释放 GPU 显存
    free(h_data);       // 释放 CPU 内存
    return 0;
}

这就是 CUDA 程序的标准套路，每个程序都是这 7 步：

CPU GPU 1. malloc (CPU 内存) 2. cudaMalloc (GPU 显存) → HBM 分配空间 3. cudaMemcpy Host→Device → 数据到达 HBM 4. kernel<<<grid, block>>>() → 线程并行执行 5. cudaMemcpy Device→Host ← 结果从 HBM 拷回 6. 验证 / 使用结果 7. cudaFree + free → 释放

1.8 Shared Memory（SRAM）

同一 block 内的线程共享，需手动管理：

__global__ void kernel() {
    __shared__ float smem[256];          // 静态分配，编译时大小确定

    smem[threadIdx.x] = global_data[...];
    __syncthreads();                     // 同步屏障：等所有线程写完再读
    // 现在可以安全读其他线程写入的 smem 数据
}
关键概念
__syncthreads() 是块内同步——保证同一 block 的所有线程都执行到这一行后才继续。不同 block 之间无法同步（这就是为什么 FlashAttention 的每个 block 独立计算自己那部分）。

1.9 矩阵在内存中的布局

矩阵是 2D 的，但内存是 1D 的。C/CUDA 默认 row-major（行优先）：

矩阵 A (3x4): ┌─────────────────┐ │ a00 a01 a02 a03 │ ← 第 0 行 │ a10 a11 a12 a13 │ ← 第 1 行 │ a20 a21 a22 a23 │ ← 第 2 行 └─────────────────┘ 内存中（连续 1D）: [a00, a01, a02, a03, a10, a11, a12, a13, a20, a21, a22, a23] ──── 第 0 行 ──── ──── 第 1 行 ──── ──── 第 2 行 ──── 访问 A[row][col] = A[row * N_cols + col] 例: A[1][2] = A[1 * 4 + 2] = A[6] = a12
1.5

热身：Vector 求和（Reduction）

在进入矩阵乘法之前，先用一个最简单的例子感受 CUDA 并行编程：把一个长度为 N 的数组所有元素求和。

思路：并行归约（Parallel Reduction）

CPU 上求和是一个 for 循环，O(N)。GPU 上可以用树形归约：每一轮让一半线程把相邻两个元素加在一起，log₂(N) 轮后得到结果。

原始数据: [1 3 5 7 2 4 6 8] 第 1 轮 (stride=1): 每个偶数位线程把相邻元素加起来 thread 0: 1+3=4 thread 2: 5+7=12 thread 4: 2+4=6 thread 6: 6+8=14 → [4 . 12 . 6 . 14 . ] 第 2 轮 (stride=2): thread 0: 4+12=16 thread 4: 6+14=20 → [16 . . . 20 . . . ] 第 3 轮 (stride=4): thread 0: 16+20=36 → [36 . . . . . . . ] ← 最终结果！

完整代码

#include <stdio.h>
#include <cuda_runtime.h>

// =============================================
// Kernel: 块内并行归约求和
// 每个 block 把自己负责的那段数组求和
// =============================================
__global__ void block_reduce_sum(const float *input, float *output, int N) {
    __shared__ float smem[256];  // block 内共享的 SRAM

    int tid = threadIdx.x;
    int gid = blockIdx.x * blockDim.x + threadIdx.x;

    // 1. 每个线程从 HBM 搬一个元素到 shared memory
    smem[tid] = (gid < N) ? input[gid] : 0.0f;
    __syncthreads();  // 等所有线程搬完

    // 2. 树形归约：每轮活跃线程减半
    for (int stride = blockDim.x / 2; stride > 0; stride /= 2) {
        if (tid < stride) {
            smem[tid] += smem[tid + stride];
        }
        __syncthreads();  // 每轮都要同步！
    }

    // 3. thread 0 持有本 block 的部分和，写到 output
    if (tid == 0) {
        output[blockIdx.x] = smem[0];
    }
}

int main() {
    int N = 1024;
    int block_size = 256;
    int grid_size = (N + block_size - 1) / block_size;  // = 4 个 block

    // --- CPU 侧初始化 ---
    float *h_input = (float *)malloc(N * sizeof(float));
    for (int i = 0; i < N; i++) h_input[i] = 1.0f;  // 全 1，预期结果 = 1024

    // --- GPU 侧分配 ---
    float *d_input, *d_partial;
    cudaMalloc(&d_input, N * sizeof(float));
    cudaMalloc(&d_partial, grid_size * sizeof(float));

    cudaMemcpy(d_input, h_input, N * sizeof(float), cudaMemcpyHostToDevice);

    // --- 第一轮：每个 block 归约 256 个元素 → 4 个部分和 ---
    block_reduce_sum<<<grid_size, block_size>>>(d_input, d_partial, N);

    // --- 第二轮：1 个 block 归约 4 个部分和 → 最终结果 ---
    float *d_result;
    cudaMalloc(&d_result, sizeof(float));
    block_reduce_sum<<<1, grid_size>>>(d_partial, d_result, grid_size);

    // --- 拷回并打印 ---
    float result;
    cudaMemcpy(&result, d_result, sizeof(float), cudaMemcpyDeviceToHost);
    printf("Sum = %.1f (expected %.1f)\n", result, (float)N);

    cudaFree(d_input); cudaFree(d_partial); cudaFree(d_result);
    free(h_input);
    return 0;
}

为什么要两轮 kernel？

跨 block 无法同步
__syncthreads() 只能同步同一 block 内的线程。不同 block 之间无法协调，所以第一轮每个 block 各自归约出一个部分和，再启动第二轮 kernel 把这些部分和归约到最终结果。
第一轮 kernel: 4 个 block，每个 block 256 个线程 Block 0: input[0..255] → partial[0] = 256 Block 1: input[256..511] → partial[1] = 256 Block 2: input[512..767] → partial[2] = 256 Block 3: input[768..1023]→ partial[3] = 256 第二轮 kernel: 1 个 block，4 个线程 Block 0: partial[0..3] → result = 1024

关键要点

  • Shared memory：block 内 256 个线程共享 smem[]，避免反复访问 HBM
  • __syncthreads()：每轮归约后必须同步，否则快线程会读到慢线程还没更新的值
  • stride 减半：第一轮 128 个线程活跃，第二轮 64 个……最后只剩 thread 0
  • 两轮 kernel：因为跨 block 无法同步，必须分两次启动
与 GEMM 的联系
Vector sum 展示了 CUDA 的核心套路：HBM → shared memory → 并行计算 → 写回。接下来的 GEMM 用的是完全相同的模式，只是每个线程做的不是加法而是内积。
02

朴素 GEMM（Naive GEMM）

GEMM 是什么？
GEMM = GEneral Matrix Multiply（通用矩阵乘法），是 BLAS（Basic Linear Algebra Subprograms）中的 Level 3 操作。完整定义是 C = α · A × B + β · C，本文简化为 α=1, β=0 的情况，即 C = A × B。

2.1 思路

C = A x B，其中 A: (M, K)，B: (K, N)，C: (M, N)。

每个线程计算 C 的一个元素——即 A 的一行和 B 的一列的内积：

C[row][col] = Σk A[row][k] * B[k][col]

线程 (row, col) 负责 C[row][col]，独立从 HBM 读 A 的第 row 行和 B 的第 col 列，做 K 次乘加。

线程 (1,2) 负责 C[1][2]: A: B: C: k=0 k=1 k=2 col=2 col=2 ↓ ↓ ↓ ↓ ↓ row=1 [a b c] x [ . . x ] → [ . . ? ] [ . . y ] row=1 [ . . z ] C[1][2] = a*x + b*y + c*z

2.2 完整代码

// naive_gemm.cu

#include <stdio.h>
#include <stdlib.h>
#include <cuda_runtime.h>

// =============================================
// Kernel: 朴素 GEMM
// 每个线程计算 C 的一个元素
// =============================================
__global__ void naive_gemm(const float *A, const float *B, float *C,
                           int M, int K, int N) {
    // 当前线程负责 C 的哪个元素
    int row = blockIdx.y * blockDim.y + threadIdx.y;  // 行
    int col = blockIdx.x * blockDim.x + threadIdx.x;  // 列

    if (row < M && col < N) {
        float sum = 0.0f;
        for (int k = 0; k < K; k++) {
            sum += A[row * K + k] * B[k * N + col];
            //      A[row][k]        B[k][col]
        }
        C[row * N + col] = sum;
    }
}

// =============================================
// Host 代码
// =============================================
void run_naive_gemm(int M, int K, int N) {
    // --- 1. 在 CPU 上分配并初始化矩阵 ---
    size_t size_A = M * K * sizeof(float);
    size_t size_B = K * N * sizeof(float);
    size_t size_C = M * N * sizeof(float);

    float *h_A = (float *)malloc(size_A);
    float *h_B = (float *)malloc(size_B);
    float *h_C = (float *)malloc(size_C);

    // 简单初始化
    for (int i = 0; i < M * K; i++) h_A[i] = (float)(rand() % 10);
    for (int i = 0; i < K * N; i++) h_B[i] = (float)(rand() % 10);

    // --- 2. 在 GPU 上分配显存 ---
    float *d_A, *d_B, *d_C;
    cudaMalloc(&d_A, size_A);
    cudaMalloc(&d_B, size_B);
    cudaMalloc(&d_C, size_C);

    // --- 3. 拷贝数据 CPU → GPU ---
    cudaMemcpy(d_A, h_A, size_A, cudaMemcpyHostToDevice);
    cudaMemcpy(d_B, h_B, size_B, cudaMemcpyHostToDevice);

    // --- 4. 配置并启动 kernel ---
    dim3 block_size(16, 16);  // 每个 block 16x16 = 256 个线程
    dim3 grid_size((N + block_size.x - 1) / block_size.x,
                   (M + block_size.y - 1) / block_size.y);

    naive_gemm<<<grid_size, block_size>>>(d_A, d_B, d_C, M, K, N);

    // --- 5. 拷回结果 GPU → CPU ---
    cudaMemcpy(h_C, d_C, size_C, cudaMemcpyDeviceToHost);

    // --- 6. 验证（打印左上角 4x4）---
    printf("C[0:4][0:4] =\n");
    for (int i = 0; i < 4 && i < M; i++) {
        for (int j = 0; j < 4 && j < N; j++)
            printf("%8.1f ", h_C[i * N + j]);
        printf("\n");
    }

    // --- 7. 清理 ---
    cudaFree(d_A); cudaFree(d_B); cudaFree(d_C);
    free(h_A); free(h_B); free(h_C);
}

int main() {
    int M = 1024, K = 512, N = 768;
    printf("Naive GEMM: (%d x %d) x (%d x %d)\n", M, K, K, N);
    run_naive_gemm(M, K, N);
    return 0;
}

2.3 性能分析：为什么慢

线程 (row=0, col=0): 从 HBM 读 A 的第 0 行（K 个元素）+ B 的第 0 列（K 个元素） 线程 (row=0, col=1): 从 HBM 读 A 的第 0 行（K 个元素）+ B 的第 1 列（K 个元素） 线程 (row=0, col=2): 从 HBM 读 A 的第 0 行（K 个元素）+ B 的第 2 列（K 个元素） ... A 的第 0 行被 N 个线程各读了一次，B 的第 0 列被 M 个线程各读了一次。 没有数据复用，全靠 HBM 带宽硬扛。 HBM 总读取量 = 每个线程读 2K 个元素 x M*N 个线程 = 2 * M * N * K
03

Tiling GEMM（分块 GEMM）

3.1 核心思想

利用 shared memory（SRAM） 实现数据复用：

  1. 把 C 矩阵分成 TILE x TILE 的小块，每个 block 负责一个 C 块
  2. 沿 K 维度分段，每段加载一个 A tile 和一个 B tile 到 shared memory
  3. block 内所有线程共享这份数据，完成小矩阵乘法后累加
  4. 遍历完所有 K 段后，写回结果
K 维度方向 → ┌────┬────┬────┬────┐ A: │ A0 │ A1 │ A2 │ A3 │ ← 第 row_block 行的 tile └────┴────┴────┴────┘ 对每个 k_step: ┌────┐ 从 HBM 加载 A_tile 和 B_tile 到 SRAM B: │ B0 │ ← 第 col_block 列 在 SRAM 中做小矩阵乘法 ├────┤ 的 tile 累加到 C_tile │ B1 │ ├────┤ C_tile = A0*B0 + A1*B1 + A2*B2 + A3*B3 │ B2 │ ├────┤ │ B3 │ └────┘

3.2 详细图解

以 TILE=16 为例，block(1,2) 负责 C[16:32, 32:48] 这个 16x16 的子矩阵：

K = 64，分成 64/16 = 4 段 Step 0 (k=0..15): A_tile = A[16:32, 0:16] ← 从 HBM 加载到 shared memory B_tile = B[0:16, 32:48] ← 从 HBM 加载到 shared memory __syncthreads() ← 等所有线程加载完毕 C_tile += A_tile x B_tile ← 在 SRAM 中计算 __syncthreads() ← 等所有线程计算完再加载下一块 Step 1 (k=16..31): A_tile = A[16:32, 16:32] B_tile = B[16:32, 32:48] C_tile += A_tile x B_tile Step 2 (k=32..47): ... Step 3 (k=48..63): ... 写回: C[16:32, 32:48] = C_tile

3.3 数据复用分析

朴素 GEMM SLOW

block 内 16x16 = 256 个线程

每个线程独立读 A 的一行（K 个元素）

同一行被 16 个线程各读一次

HBM 读取: 256 x 2K = 512K

Tiling GEMM FAST

每个 k_step，block 协作加载：

A_tile: 16x16 = 256 个元素

B_tile: 16x16 = 256 个元素

共 K/16 步，总 HBM 读取: 32K

节省倍数
512K / 32K = 16 = TILE。A_tile 加载 1 次，被 16 列的线程共享；B_tile 加载 1 次，被 16 行的线程共享。数据复用 TILE 倍。

3.4 完整代码

// tiling_gemm.cu

#include <stdio.h>
#include <stdlib.h>
#include <cuda_runtime.h>

#define TILE 16

// =============================================
// Kernel: Tiling GEMM
// 每个 block 计算 C 的一个 TILE x TILE 块
// =============================================
__global__ void tiling_gemm(const float *A, const float *B, float *C,
                            int M, int K, int N) {
    // --- 分配 shared memory ---
    __shared__ float As[TILE][TILE];  // A 的 tile
    __shared__ float Bs[TILE][TILE];  // B 的 tile

    // 当前线程在 block 内的位置
    int tx = threadIdx.x;  // 列
    int ty = threadIdx.y;  // 行

    // 当前线程负责的 C 元素的全局位置
    int row = blockIdx.y * TILE + ty;
    int col = blockIdx.x * TILE + tx;

    float sum = 0.0f;

    // --- 沿 K 维度分段遍历 ---
    for (int k_step = 0; k_step < K; k_step += TILE) {
        // 协作加载：每个线程搬一个元素到 shared memory
        // 256 个线程，刚好搬 16x16 = 256 个元素
        if (row < M && (k_step + tx) < K) {
            As[ty][tx] = A[row * K + k_step + tx];
        } else {
            As[ty][tx] = 0.0f;  // 越界补零
        }

        if ((k_step + ty) < K && col < N) {
            Bs[ty][tx] = B[(k_step + ty) * N + col];
        } else {
            Bs[ty][tx] = 0.0f;
        }

        // 同步：确保 tile 加载完毕
        __syncthreads();

        // 在 shared memory 中做小矩阵乘法
        for (int k = 0; k < TILE; k++) {
            sum += As[ty][k] * Bs[k][tx];
            //     ↑ 同一行线程共享 As 同一行（A 的复用）
            //              ↑ 同一列线程共享 Bs 同一列（B 的复用）
        }

        // 同步：确保所有线程用完当前 tile 后再加载下一个
        __syncthreads();
    }

    // 写回结果
    if (row < M && col < N) {
        C[row * N + col] = sum;
    }
}

// =============================================
// Host 代码
// =============================================
void run_tiling_gemm(int M, int K, int N) {
    size_t size_A = M * K * sizeof(float);
    size_t size_B = K * N * sizeof(float);
    size_t size_C = M * N * sizeof(float);

    float *h_A = (float *)malloc(size_A);
    float *h_B = (float *)malloc(size_B);
    float *h_C = (float *)malloc(size_C);

    for (int i = 0; i < M * K; i++) h_A[i] = (float)(rand() % 10);
    for (int i = 0; i < K * N; i++) h_B[i] = (float)(rand() % 10);

    float *d_A, *d_B, *d_C;
    cudaMalloc(&d_A, size_A);
    cudaMalloc(&d_B, size_B);
    cudaMalloc(&d_C, size_C);

    cudaMemcpy(d_A, h_A, size_A, cudaMemcpyHostToDevice);
    cudaMemcpy(d_B, h_B, size_B, cudaMemcpyHostToDevice);

    dim3 block_size(TILE, TILE);
    dim3 grid_size((N + TILE - 1) / TILE,
                   (M + TILE - 1) / TILE);

    tiling_gemm<<<grid_size, block_size>>>(d_A, d_B, d_C, M, K, N);

    cudaMemcpy(h_C, d_C, size_C, cudaMemcpyDeviceToHost);

    printf("C[0:4][0:4] =\n");
    for (int i = 0; i < 4 && i < M; i++) {
        for (int j = 0; j < 4 && j < N; j++)
            printf("%8.1f ", h_C[i * N + j]);
        printf("\n");
    }

    cudaFree(d_A); cudaFree(d_B); cudaFree(d_C);
    free(h_A); free(h_B); free(h_C);
}

int main() {
    int M = 1024, K = 512, N = 768;
    printf("Tiling GEMM (TILE=%d): (%d x %d) x (%d x %d)\n", TILE, M, K, K, N);
    run_tiling_gemm(M, K, N);
    return 0;
}

3.5 两次 __syncthreads() 的必要性

for (int k_step = 0; k_step < K; k_step += TILE) {
    // 加载到 shared memory
    As[ty][tx] = A[...];
    Bs[ty][tx] = B[...];

    __syncthreads();    // ① 加载屏障：等所有线程写完 shared memory

    for (int k = 0; k < TILE; k++)
        sum += As[ty][k] * Bs[k][tx];  // 读其他线程写入的数据

    __syncthreads();    // ② 计算屏障：等所有线程用完当前 tile
}
屏障 №1 — 加载屏障
线程 A 可能还没写完 As[3][5]，线程 B 就去读了——读到旧值或未初始化值。
屏障 №2 — 计算屏障
快线程进入下一轮循环，开始写新的 As[ty][tx]，而慢线程还在用旧的 As[ty][k] 做内积——数据被覆盖，结果错误。
时间线（无屏障 ② 的错误情况）: 快线程: 加载 step0 → 计算 step0 → 加载 step1（覆盖 As！） 慢线程: 加载 step0 → 计算 step0（还在读 As，但已被覆盖）→ 💥

06

标准 Attention 的问题

标准 Attention 公式：O = softmax(Q · KT / √d) · V，在 PyTorch 中通常拆成 3 个独立 kernel：

S = Q @ K.T          # kernel 1: 矩阵乘法（cuBLAS GEMM）
P = torch.softmax(S) # kernel 2: softmax
O = P @ V            # kernel 3: 矩阵乘法

6.1 为什么慢：Kernel 间的 HBM 往返

每个 kernel 独立调度，kernel 之间无法共享 SRAM——SRAM 绑定在线程块上，kernel 结束时线程块销毁，SRAM 随之释放。

3 个 Kernel（标准 Attention）: HBM → SRAM → 算 S → 写回 HBM → SRAM → softmax → 写回 HBM → SRAM → 乘 V → HBM ↑ 中间矩阵 S ↑ 中间矩阵 P O(N²) 显存 O(N²) 显存
为什么中间结果必须写回 HBM？
SRAM 就像函数的局部变量——函数返回后栈空间就释放了。Kernel 1 结束后 S 矩阵在 SRAM 中的数据就丢了，Kernel 2 必须从 HBM 重新读取。不是因为想存，而是不存就丢了。
Kernel 的生命周期: Kernel 1 启动 → 分配线程块到各 SM → 每个线程块获得一块 SRAM ↓ 计算完毕，结果写入 HBM ↓ Kernel 1 结束 → 线程块销毁 → SRAM 释放（数据丢失） ↓ Kernel 2 启动 → 分配新的线程块 → 获得全新的 SRAM → 从 HBM 重新读数据

6.2 Compute-bound vs Memory-bound

类型瓶颈典型操作
Compute-bound算力（FLOPs）大矩阵乘法、大卷积
Memory-bound带宽（HBM 读写）softmax、dropout、LayerNorm 等逐元素操作

Attention 中的 softmax、masking、dropout 都是 memory-bound——计算量很小，但每个元素都要从 HBM 读一次、写一次。这意味着减少 HBM 访问次数比减少 FLOPs 更重要。

6.3 标准 Attention 的开销

项目值
额外显存O(N²)（存 S 和 P 两个 N×N 矩阵）
HBM IOO(N² + Nd)（主要是读写 S、P 各一次）
计算量O(N²d)

N 大时，O(N²) 的 S、P 矩阵占满显存，且读写它们成为 IO 瓶颈。

07

Kernel Fusion（算子融合）

解决 memory-bound 问题的标准思路：把多个操作融合成一个 kernel，数据在 SRAM 上流转，避免反复写回/读取 HBM。

不融合 3 个 Kernel

HBM → 算 S → 写回 HBM

HBM → softmax → 写回 HBM

HBM → 乘 V → HBM

中间结果反复落地 HBM

融合 1 个 Kernel

HBM → SRAM

算 S → softmax → 乘 V

全在 SRAM 上完成

→ HBM

中间不碰 HBM

FlashAttention 本质上就是一个手写的融合 CUDA kernel，把 attention 的全部操作（matmul + mask + softmax + dropout + matmul）融合在一起，配合 tiling 让数据块刚好放进 SRAM。

08

Online Softmax

融合成一个 kernel 后有个数学难题：softmax 需要全局 max 和 sum，但分块后每块只看到局部数据。Online Softmax 解决了这个问题。

8.1 标准 Softmax 需要 3-pass

Safe Softmax（3 pass）: Pass 1: 遍历 x，求 m = max(x) ← 需要看完所有元素 Pass 2: 遍历 x，求 l = Σ exp(x_j - m) ← 又遍历一次 Pass 3: 遍历 x，求 softmax(x_j) = exp(x_j - m) / l ← 再遍历一次

8.2 递推公式推导

已知处理完前 i-1 个元素后：

di-1 = Σj=1i-1 exp(xj - mi-1),  mi-1 = max(x1, ..., xi-1)

现在来了第 i 个元素，max 可能变了：mi = max(mi-1, xi)。

问题：di-1 里用的是旧 max，需要换成新 max。关键洞察——修正因子不依赖 j：

exp(xj - mold) × exp(mold - mnew) = exp(xj - mnew)

所以可以一次性把整个 di-1 乘上修正因子：

Online Softmax 递推公式
mi = max(mi-1, xi)
di = di-1 × exp(mi-1 - mi) + exp(xi - mi)
把 3-pass 的前两步（求 max + 求 sum）合并成1-pass。

8.3 Online Softmax → 2-pass（极限）

Online Softmax（2 pass）: Pass 1: 遍历 x，同时求 m 和 l ← 合并了 Pass 1+2 Pass 2: 遍历 x，求 softmax(x_j) ← 无法省掉，必须等 m 和 l 确定 Pass 2 无法省掉——要算出每个元素的 softmax 值，必须等 m 和 l 都确定。 对纯 softmax而言，2-pass 是极限。

8.4 FlashAttention：进一步到 1-pass

FlashAttention 能做到 1-pass，因为 attention 的目标不是 softmax 本身，而是加权和 O = softmax(s) · V。分子 o 可以和分母 d 用相同的修正因子同步递推：

Attention 的在线递推
mi = max(mi-1, si)
di = di-1 × exp(mi-1 - mi) + exp(si - mi)
oi = oi-1 × exp(mi-1 - mi) + exp(si - mi) × vi

最终: O = oN / dN
分子 o 和分母 d 独立累加，最后一除即可。不需要单独算出每个 softmax 值。

8.5 数值例子

N=3，d=2，scores = [2, 4, 3]，values = [[10,1], [20,2], [30,3]]：

标准做法（等全部算完再乘）: m = 4, d = exp(-2) + 1 + exp(-1) ≈ 1.503 w1 ≈ 0.090, w2 ≈ 0.665, w3 ≈ 0.245 o = 0.090×[10,1] + 0.665×[20,2] + 0.245×[30,3] = [21.55, 2.155] Online 做法（边扫描边累加）: j=1: m=2, d=1, o=[10, 1] j=2: m=4, d=1.135, o=[1.35,0.135]+[20,2] = [21.35, 2.135] j=3: m=4, d=1.503, o=[21.35,2.135]+[11.04,1.104] = [32.39, 3.239] 最终归一化: O = [32.39, 3.239] / 1.503 = [21.55, 2.155] ✓
09

FlashAttention 算法

结合Tiling + Online Softmax + Kernel Fusion，FlashAttention 把 Q 分成 Tr 块、K/V 分成 Tc 块，块大小由 SRAM 容量决定。

9.1 伪代码

# Q 分成 T_r = ceil(N / B_r) 个块, K/V 分成 T_c = ceil(N / B_c) 个块
# B_r, B_c 选择使得 Q块 + K块 + V块 + 中间变量 ≤ SRAM 大小 M

O = zeros(N, d)

for i in range(T_r):                            # 外层遍历 Q 的块
    Q_i = load_from_HBM(Q[i*B_r : (i+1)*B_r])   # 加载 Q 块到 SRAM: (B_r, d)
    O_i = zeros(B_r, d)                          # O 在 SRAM 中初始化
    m_i = full(B_r, -inf)                        # running max，在 SRAM 中
    d_i = zeros(B_r)                             # running sum，在 SRAM 中

    for j in range(T_c):                      # 内层遍历 K/V 的块
        K_j = load_from_HBM(K[j*B_c : (j+1)*B_c])
        V_j = load_from_HBM(V[j*B_c : (j+1)*B_c])

        # --- 以下全在 SRAM 中计算 ---
        S_block = Q_i @ K_j.T               # (B_r, B_c) 小矩阵，放得下

        m_new = max(m_i, rowmax(S_block))
        correction = exp(m_i - m_new)        # online softmax 修正因子
        P_block = exp(S_block - m_new)

        d_i = d_i * correction + rowsum(P_block)
        O_i = O_i * correction + P_block @ V_j
        m_i = m_new
        # --- SRAM 计算结束 ---

    O_i = O_i / d_i                              # 归一化，仍在 SRAM 中
    store_to_HBM(O[i*B_r : (i+1)*B_r], O_i)     # O_i 只写回 HBM 1 次！

9.2 和 Tiling GEMM 的对比

Tiling GEMM

每个 block 计算 C 的一块

C_tile = Σk A_tile_k × B_tile_k

纯矩阵乘法，累加就行

FlashAttention

每个 block 计算 O 的一块

O = softmax(Q·KT) × V

需要 online softmax 处理跨块归一化

核心联系
FlashAttention 的分块循环结构和 tiling GEMM 的 k_step 循环是同构的，只是每一步多了 softmax 的在线修正。

9.3 Tile 大小由 SRAM 决定

SRAM 要同时放下 Q_tile、K_tile、V_tile 和中间变量：

以 A100（SRAM ~192 KB/SM）、float16（2 bytes）为例： TILE=64: 3 × 64² × 2 = 24 KB ← 放得下，常用选择 TILE=128: 3 × 128² × 2 = 96 KB ← 勉强 TILE=256: 3 × 256² × 2 = 384 KB ← 超出，放不下

9.4 复杂度对比

标准 AttentionFlashAttention

HBM IOO(N² + Nd)O(N²d² / M)
计算量O(N²d)O(N²d)（不变）

M = SRAM 大小。SRAM 越大，IO 越少——IO 复杂度中 M 在分母。

9.5 循环顺序：从 v1 到 v2

FlashAttention-1 采用外层 K/V、内层 Q，FlashAttention-2 改为外层 Q、内层 K/V（上面伪代码所示）。渐近复杂度相同，但外层 Q 更优：

外层 K/V，内层 Q（FlashAttention-1）: for j in K/V blocks: ← K_j 加载 1 次 for i in Q blocks: O_i = read_from_HBM() ← 每轮都要从 HBM 读 O_i O_i = update(...) write_to_HBM(O_i) ← 每轮都要写回 O_i → O_i 被读写 T_c 次！ 外层 Q，内层 K/V（FlashAttention-2）: for i in Q blocks: ← Q_i 加载 1 次，O_i 在 SRAM 中初始化 for j in K/V blocks: K_j, V_j = load(...) O_i = update(...) ← O_i 一直留在 SRAM 里累加 write_to_HBM(O_i) ← O_i 只写回 1 次！
为什么 v1 选外层 K/V？
不是因为前向传播更好，而是反向传播的实现考虑——反向传播需要对 K/V 做梯度累加，外层 K/V 更容易避免多个线程块对同一位置的原子写冲突。FlashAttention-2 解决了这个工程问题后，前向传播改为外层 Q，速度提升约 2 倍。

Flash Decoding

10.1 问题：Decode 阶段 GPU 利用率低

FlashAttention 沿 Q 的 seq_len 分块获得并行度。但 decode 阶段每步只产生 1 个新 token，Q 只有 1 行：

Prefill: Q = (batch, n_heads, 数千行, head_dim) → 充分并行 Decode: Q = (batch, n_heads, 1 行, head_dim) → 退化为 1 块 典型场景 batch=1, heads=32: 并行度 = 32 个线程块 A100 有 108 个 SM → 大量 SM 闲置，利用率 ~30%

10.2 核心思路：沿 KV 的 seq_len 并行

Q 没得分，那就分 KV——把 KV cache 沿序列维度切成 S 块，每块独立计算局部注意力，最后合并。

Flash Decoding: 将 KV cache 拆成 S 块 Block 0: q × K[0:32]T → local softmax → × V[0:32] → partial_o_0, lse_0 Block 1: q × K[32:64]T → local softmax → × V[32:64] → partial_o_1, lse_1 ... Block S-1: q × K[...] → ... → partial_o_{S-1}, lse_{S-1} Reduction（用 log-sum-exp 合并）: global_lse = log(Σ exp(lse_i)) weight_i = exp(lse_i - global_lse) final_o = Σ weight_i × partial_o_i

10.3 为什么合并是正确的

每块的局部 softmax 分母占全局的比例就是该块的合并权重：

weighti = exp(lsei) / Σk exp(lsek) = exp(lsei - global_lse)

本质和 online softmax 的修正因子一样——分子分母独立累加，最后一除。

10.4 并行度对比

FlashAttention (decode)Flash Decoding
并行维度batch × n_headsbatch × n_heads × S
典型值 (batch=1, heads=32, S=128)32 个线程块4096 个线程块
A100 SM 利用率 (108 SMs)~30%~100%

常见问题

<<<grid_size, block_size>>> 的第一个参数是什么？

是的，第一个参数 grid_size 指的就是 Grid 中有多少个 Block。

参数含义说明
grid_size（第一个）Grid 中有多少个 Block即 block 的数量

总线程数 = grid_size x block_size
