Tree Draft 基础：静态树、动态树与工程约束

这篇文档面向工程实现，说明 tree draft 的基本概念、静态树和动态树的定义、为什么当前通常不先做静态 tree、为什么动态 tree 很难与 CUDA graph 兼容，以及为什么在高并发服务场景下 speculative decoding 的收益会变小。

先记核心结论：
  1. tree draft 是让 draft model 生成一棵候选 token 树，而不是一条候选链。
  2. 静态 tree 指树骨架固定；动态 tree 指树骨架运行时根据 logits / score 决定。
  3. 静态 tree 只解决“树形是否固定”的问题，并不能自动解决 tree attention mask 这个根问题。
  4. 动态 tree 的难点不只是 mask，还包括 shape 变化、控制流变化、path compaction、CPU/GPU overlap，所以很难吃到 CUDA graph。
  5. 高并发时 speculative decoding 收益有限，核心原因是 continuous batching 已经把 target decode 的串行空洞填掉了。

1. 什么是 tree draft

普通 speculative decoding 里，draft model 通常只生成一条候选链： 

context -> t1 -> t2 -> t3

tree draft 则不是一条链，而是一棵候选树： 

context
├─ A
│  ├─ B
│  └─ C
└─ D
   ├─ E
   └─ F

最终 target model 并不会把整棵树全部接受，而是一次性验证整棵树，然后从中接受一条 path（或 path 的前缀）。

tree draft 的直觉： 线性 draft 的问题是“第一个 token 猜错，后面整条链都废了”；tree draft 试图用更多候选分支来提高 target 的命中率。

1.1 为什么 tree draft 需要专门的 attention

树节点通常要被压平成一维 token 序列再送入 verifier。例如上面的树可以被 flatten 成： 

[A, D, B, C, E, F]

但压平之后，不能直接使用普通 causal mask。因为例如 B 只能看：

  • 原始 context
  • 自己的祖先 A
  • 自己

它不能看到兄弟分支上的 D、E、F。所以 tree draft 需要的是 ancestor mask / tree attention，而不是普通的 causal attention。

2. 静态 tree draft 与动态 tree draft

2.1 静态 tree draft 的定义

静态 tree 的意思是：树的骨架在推理前就固定了。固定的是：

  • 最大深度
  • 每层最多多少节点
  • 每个节点有哪些子节点位置
  • flatten 顺序和 parent / ancestor 关系

注意，固定的是结构，不是 token 内容。每个节点最终填什么 token，仍然是运行时根据 draft logits 得到的。 

静态 tree 模板（示意）
root
├─ n1
│  ├─ n3
│  └─ n4
└─ n2
   ├─ n5
   └─ n6

2.2 动态 tree draft 的定义

动态 tree 的意思是：只有预算固定，树怎么长由运行时决定。例如：

  • 当前 top-1 概率很高，就沿着一条路继续往深处扩
  • 当前分布很平，就多开几个兄弟分支
  • 某条 path 分数更高，就把更多节点预算给它
同样是 8 个节点预算

场景 A（模型很确定）:
root -> A -> B -> C -> D

场景 B（模型不确定）:
root
├─ A
├─ B
├─ C
└─ D
维度 静态 tree 动态 tree
树形 预先固定 运行时决定
节点预算分配 按模板预留 按当前 logits / score 分配
parent / ancestor 关系 固定 每轮变化
verifier 输入布局 flatten 模板固定 每轮可能重排
工程复杂度 中等 很高

3. 目前不支持静态 tree 的原因

可以压缩成 3 个主因：

  1. 现有 speculative pipeline 主要按线性链设计。 很多默认假设都是 [batch, K] 的链式 proposal、普通 causal attention，以及 verify 后只返回少量 accepted / rejected 摘要；换成 tree 后，这些假设都会失效。
  2. 静态 tree 也绕不过 tree attention。 树状 token 压成一维后，需要的是 ancestor mask，而不是普通 causal mask。标准 FlashAttention / paged causal attention 一般不直接支持这种不规则 mask，所以还需要额外的 tree-aware attention backend。
  3. verify 后的后处理仍然是动态的。 即使树模板固定，系统还是要知道最终接受的是哪条 path，并据此做 compaction、KV / metadata 更新以及下一轮输入整理，所以实现不会像线性 speculative decoding 那样简单。
一句话记忆： 静态 tree 解决的是“树形固定”，但没有解决 tree mask 和 path-based 后处理 这两个核心工程问题。

4. 为什么动态 tree 很难 CUDA graph

主因很简单：CUDA graph 需要执行图尽量固定，而动态 tree 每轮都会改执行图。 动态 tree 会让节点数、flatten 布局、tree mask、accepted path 以及后续 compaction 都随当前 logits 改变，导致输入 shape 和控制流都不稳定，graph 很难稳定 replay。

一句话记忆： 静态 tree 的主要问题是 tree mask 不规则；动态 tree 则在此基础上又多了 shape 不固定 + 控制流不固定，所以更难和 CUDA graph 兼容。

理论上可以通过很多模板、分桶和 padding 去硬做，但通常会带来模板数爆炸、预热成本高、无效计算增加，工程上往往不划算。

5. 为什么高并发下 spec 收益有限

5.1 spec 真正在优化什么

speculative decoding 的核心价值是：减少 target model 的串行 decode 步数。当请求很少时，单个请求一步一步 decode，GPU 往往吃不满；spec 通过 draft + verify，一次确认多个 token，就能明显减少 target 的串行轮数。

5.2 高并发时，continuous batching 已经在填这个坑

在高并发服务里，很多请求同时 decode。调度器会把不同请求的 token 混在一起执行，使 target model 的 decode batch 本来就很满。这时：

  • GPU 本来就接近饱和
  • 单请求的串行空洞，已经被跨请求 batching 填掉
  • spec 再减少一点 target 步数，边际收益就小了
低并发：
[target step] ... 空
[target step] ... 空
[target step] ... 空

高并发：
[target(req1)][target(req2)][target(req3)][target(req4)] ...

5.3 高并发下 spec 还会引入额外成本

额外成本 说明
draft model forward 即使 draft 很小，也不是零成本。
verify / reject-sample 需要额外的 logits 处理、接受/拒绝逻辑。
metadata 更新 slot mapping、positions、query layout 都要维护。
CPU/GPU 协同 高负载下更难把 overlap 做得漂亮。
batch token budget 挤占 draft / verify 会消耗 token budget，可能影响调度器吞吐。

5.4 decode 往往是 memory-bound，不是纯算力不够

高并发 decode 的瓶颈很多时候是 KV cache 访问和 HBM 带宽，而不是纯 FLOPs 不够。spec 并不会神奇地消灭所有内存访问，有时只是把“target 的串行步数减少了一些”，但又引入了额外 draft / verify 访问，最终吞吐提升并不大。

所以更准确的说法是： speculative decoding 不是高并发下一定没用，而是在 throughput-first、高 batch occupancy 的场景里，边际收益通常显著下降。

5.5 一个简单判断标准

场景 spec 的价值
低并发 / 单请求 latency 优先 通常更值
高并发 / 吞吐优先 通常收益变小，甚至不如直接把 target batch 做满
已有很强的 continuous batching spec 的边际收益进一步下降

6. 总结

Tree draft Static tree Dynamic tree CUDA graph High concurrency

1. tree draft 的本质是让 draft model 生成候选树，再由 target 一次性验证整棵树并接受其中一条 path。

2. 静态 tree 固定的是骨架，不是 token；但即使树形固定，仍然需要 tree-aware attention backend 来正确执行 ancestor mask。

3. 目前不支持静态 tree，主因是现有 pipeline 主要按线性链设计，而 tree 还需要额外的 tree attention 和 path-based 后处理。

4. 动态 tree 很难和 CUDA graph 兼容，主因是它会让 shape 和控制流都随运行时结果变化。

5. 高并发场景下 spec 收益有限，因为 continuous batching 已经把 target decode 的串行空洞填掉了；此时 spec 带来的额外 draft / verify / 调度成本更容易吞掉收益。

如果后续要在工程里真正推进 tree draft，最现实的顺序通常是：先定义固定 tree template，再解决 tree attention backend，再处理 verifier path compaction，最后再考虑更激进的动态 tree。
