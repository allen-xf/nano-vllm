# AI Infra 方向选择 Q&A

## RL Infra vs 推理 Infra vs Sandbox

### 对比总结

| 维度 | 推理 Infra | RL Infra | Sandbox |
|------|-----------|----------|---------|
| 市场需求 | 极高，所有AI公司都需要 | 增长快但集中在头部lab | 中高，随Agent起步 |
| 技术壁垒 | 高（系统优化、kernel、调度） | 极高（训练+推理+调度交叉） | 中（虚拟化/容器/安全） |
| 人才稀缺度 | 稀缺但竞争者多 | 极稀缺 | 低竞争（目前） |
| 稳定性 | ★★★★★ | ★★★ | ★★★★ |
| 辛苦程度 | 中 | 高（混沌、调试地狱） | 中 |
| oncall压力 | 引擎优化极低/serving高 | 高 | 中 |
| 天花板 | 高，商业化清晰 | 极高但依赖范式 | 有限，可能被云厂商吃掉 |
| 被替代风险 | 低 | 中高（范式可能变） | 中（云厂商） |
| AI技术深度 | 高 | 极高 | 低 |

### Online RL 是什么

Online RL 指模型在训练过程中实时生成样本，再用这些样本的反馈来更新自己。

核心流程：
1. 当前模型对 prompt 生成多条回复 (rollout/sampling)
2. 对生成的回复打分 (reward model / verifier / 环境反馈)
3. 用打分结果计算优势，更新策略 (PPO / GRPO / REINFORCE)
4. 回到第1步，用更新后的模型继续生成

对 Infra 的挑战：训练和推理交织、资源调度复杂、吞吐要求高、异步流水线。

### RL Infra 的风险

1. **范式替代风险** — test-time compute、蒸馏/合成数据、RLAIF简化等可能替代 online RL
2. **岗位集中度风险** — 只有头部公司招，跳槽选择面窄
3. **技术绑定风险** — 和特定训练框架强绑定，框架废弃经验贬值
4. **周期性风险** — RL 在 AI 领域经历过多次冷却期
5. **可验证场景有限** — 主要在数学、代码等可验证领域效果好

### 推理 Infra 子方向辛苦程度

| 子方向 | 辛苦程度 | 故障压力 |
|--------|----------|----------|
| 在线推理服务（serving） | 高 | 高（直面用户流量） |
| 离线批量推理（batch inference） | 中 | 低 |
| 推理引擎/kernel 优化 | 中 | 极低（不直接上线） |

### 推理引擎优化的 oncall 情况

- 大厂（字节、阿里等）：引擎组和serving组分开，引擎组基本不 oncall
- 中型公司：可能要兼顾一些部署，偶尔 oncall
- 创业公司：啥都干

### 结论

如果追求稳定、不太辛苦、少故障：**推理引擎/分布式推理优化**是最优选择。
- 数据分布式系统背景可直接复用（分布式通信、调度、内存管理）
- 不碰线上服务 = 没有oncall地狱
- 方向极稳，不依赖单一范式
- 产出可量化，晋升友好
