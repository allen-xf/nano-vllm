# GEMM 优化笔记：Register Tiling vs 基础 Tiling

## 核心区别

两者都用了 shared memory，但**计算方式完全不同**。

## 基础 Tiling（每线程算 1 个输出）

```
Block (32×32 = 1024 线程) → 负责 C 的 32×32 区域
每线程: 1 个累加器，沿 K 循环做点积

每个 tile step:
  从 SMEM 读: 1 行 A + 1 列 B → 32+32 = 64 reads (被 1024 线程共享)
  计算: 32 次 FMA（一个内积的一段）

每线程：读多，算少
```

## Register Tiling（每线程算 8×8 = 64 个输出）

```
Block (16×16 = 256 线程) → 负责 C 的 128×128 区域
每线程: 64 个累加器，每步做 8×8 外积

每个 tile step 中的每个 bk 迭代:
  从 SMEM 读: 8 个 A + 8 个 B = 16 reads
  计算: 8×8 = 64 次 FMA

每线程：读少，算多（4× 计算/访存比）
```

## 关键指标对比

| | 基础 Tiling (TILE=32) | Register Tiling |
|--|----------------------|-----------------|
| 每线程输出元素 | **1** | **64** (8×8) |
| 寄存器累加器 | 1 个 | 64 个 |
| 每次 SMEM 读取后的计算 | 1 FMA | **64 FMA** |
| Block 负责 C 区域 | 32×32 | 128×128 |
| 全局→SMEM 计算密度 | 8 FLOP/B | **32 FLOP/B** (4×) |
| SMEM→寄存器 复用率 | 低 | 高（外积模式） |

## 核心优势：外积 vs 内积

```
基础 Tiling（内积模式）:
  C[i][j] += A[i][k] * B[k][j]     ← 每次只算 1 个输出的一部分
  ↓
  一个元素的 A 值只被用 1 次

Register Tiling（外积模式）:
  for tm in 0..7:
    for tn in 0..7:
      C[tm][tn] += A[tm] * B[tn]   ← A[tm] 被用了 8 次！B[tn] 也被用了 8 次！
  ↓
  从 SMEM 读一次，计算用 8 次 → 极大减少 SMEM 带宽压力
```

## 计算密度公式

```
全局内存计算密度（与 BK 无关）：
  intensity = BM × BN / (2 × (BM + BN))

基础 Tiling (BM=BN=32):   32×32 / (2×64)   =  8 FLOP/Byte
Reg Tiling  (BM=BN=128):  128×128 / (2×256) = 32 FLOP/Byte

A10 平衡点：31.2 TFLOPS / 600 GB/s = 52 FLOP/Byte
要达到 52：需要 BM=BN≥208，实际取 256
```

## 对齐设计（BK=32）

```
BK=32 对齐 warp size，加载 A 时：

tid 0:  A[row_0][k_step + 0]
tid 1:  A[row_0][k_step + 1]
...
tid 31: A[row_0][k_step + 31]  ← 一个 warp，32 个连续 float
─────────────────────────────── = 1 次 128B 事务（完美 coalesced）

若 BK=8（未对齐）：
tid 0~7:   A[row_0][k+0..7]   ← 32B
tid 8~15:  A[row_1][k+0..7]   ← 跳行！
tid 16~23: A[row_2][k+0..7]   ← 跳行！
tid 24~31: A[row_3][k+0..7]   ← 跳行！
─────────────────────────────── = 4 次事务（浪费 3/4 带宽）
```

## 一句话总结

- 基础 tiling 解决了 **Global Memory → Shared Memory** 的瓶颈
- Register tiling 进一步解决了 **Shared Memory → 寄存器** 的瓶颈——让每次读取的数据在寄存器中被反复复用，把 ALU 真正喂饱

# Compute Bound vs Memory Bound 详解

## GPU 执行模型

GPU 执行一个 kernel 时，有两件事在**同时**发生（流水线并行）：

1. **搬数据**：从显存读取数据到计算单元
2. **算数据**：计算单元做浮点运算

```
时间线（理想流水线）：
  显存控制器：[搬第1批] [搬第2批] [搬第3批] [搬第4批] ...
  计算单元：       [算第1批] [算第2批] [算第3批] ...
```

**但前提是两者耗时一样。** 谁跑得慢谁就拖后腿。

## Memory Bound（内存瓶颈）

搬运比计算慢，计算单元在等数据：

```
显存：[====搬1====] [====搬2====] [====搬3====]
计算：     [算1]         [算2]         [算3]
              ↑空闲         ↑空闲         ↑空闲

特征：GPU 算力没用满，大部分时间在等数据
例子：向量加法 C[i] = A[i] + B[i]（读 2 个，只算 1 次）
```

## Compute Bound（计算瓶颈）

计算比搬运慢，数据堆积在缓冲区：

```
显存：[搬1] [搬2] [搬3]
计算：[========算1========] [========算2========]
          数据早搬来了，但算不完，带宽有富余

特征：GPU 带宽没用满，大部分时间在做计算
这其实是好事！说明计算单元在满负荷工作
```

## 平衡点（Arithmetic Intensity Balance Point）

平衡点是 GPU 的一个硬件特性参数：

```
平衡点 = 峰值算力 / 峰值带宽

A10:  31.2 TFLOPS / 600 GB/s = 52 FLOP/Byte
4090: 82.6 TFLOPS / 1008 GB/s = 82 FLOP/Byte
H100: 989 TFLOPS(TF32) / 3350 GB/s = 295 FLOP/Byte
```

意思是：**每从显存读 1 Byte 数据，GPU 能做 52 次浮点运算（A10）。**

## 用平衡点判断瓶颈

对于你的 kernel，算一下它的**计算密度**：

```
计算密度 = kernel 总计算量 (FLOPs) / kernel 总访存量 (Bytes)
```

| 情况 | 含义 | 瓶颈 |
|------|------|------|
| 计算密度 **<** 52 | 数据搬得慢，算力在等 | **Memory Bound** |
| 计算密度 **>** 52 | 数据搬运跟得上，但算不完 | **Compute Bound** |
| 计算密度 **=** 52 | 搬运和计算同时结束 | **平衡点** |

## Roofline Model 图示

```
性能
(FLOPS)
  │          ╱ ─ ─ ─ ─ ─ ─ ─ ─ ─  ← 算力天花板 (31.2 TFLOPS)
  │        ╱
  │      ╱
  │    ╱  ← 带宽斜坡（斜率 = 600 GB/s）
  │  ╱
  │╱
  └──────────┬──────────────────── 计算密度 (FLOP/Byte)
             52
          (平衡点)

  左侧 = Memory Bound（性能被带宽限制）
  右侧 = Compute Bound（性能被算力限制）
```

## 为什么 Compute Bound 其实是目标？

| | Memory Bound | Compute Bound |
|--|-------------|---------------|
| 算力利用率 | 低（在等数据） | **高（满载）** |
| 带宽利用率 | **高（满载）** | 低（数据早到了） |
| 优化方向 | 减少访存 / 增加复用 | 换更快的 GPU / 用 Tensor Core |

**Compute Bound 反而是好状态！** 意味着 GPU 算力被完全压榨，瓶颈在硬件极限而非代码低效。

## 为什么搬运比计算慢？数字说话

```
搬 1 Byte 耗时：1 / 600 GB/s = 1.67 ns
做 52 次运算耗时：52 / 31.2 TFLOPS = 1.67 ns ← 平衡点

基础 Tiling（计算密度 8 FLOP/B）：
  搬 1 Byte: 1.67 ns
  算 8 次:   0.26 ns  ← 0.26ns 就算完了，剩下 1.41ns 在等数据！
  GPU 算力利用率 = 0.26 / 1.67 = 15%

Register Tiling（计算密度 32 FLOP/B）：
  搬 1 Byte: 1.67 ns
  算 32 次:  1.03 ns  ← 更好！但还有 0.64ns 在等
  GPU 算力利用率 = 1.03 / 1.67 = 62%
```

## GEMM 各版本对比

| Kernel | 计算密度 | vs A10 平衡点 | 算力利用率 | 状态 |
|--------|---------|--------------|-----------|------|
| Naive GEMM | <1 | 远低 | ~2% | 严重 Memory Bound |
| 基础 Tiling | 8 | 远低 | ~15% | Memory Bound |
| Register Tiling | 32 | 接近 | ~62% | 轻度 Memory Bound |
| cuBLAS 级别 | 50+ | ≈ 平衡 | 95%+ | 接近 Compute Bound |

## 一句话总结

优化的本质：把 kernel 从 Roofline 的左侧（带宽斜坡）推向右侧（算力天花板），让搬数据不再拖后腿——把瓶颈从内存推到计算，意味着硬件被完全榨干了。
